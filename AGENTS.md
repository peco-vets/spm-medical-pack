# Everything Claude Code (ECC) — エージェント指示書

これは**本番運用可能な AI コーディングプラグイン**であり、ソフトウェア開発向けに 60 の専門エージェント、232 のスキル、75 のコマンド、自動化フックワークフローを提供する。

**Version:** 2.0.0-rc.1

## 基本原則

1. **エージェント優先** — ドメイン固有のタスクは専門エージェントに委任する
2. **テスト駆動** — 実装前にテストを書き、80% 以上のカバレッジを必須とする
3. **セキュリティ優先** — セキュリティを妥協しない。すべての入力を検証する
4. **イミュータビリティ** — 既存オブジェクトを変更せず、常に新しいオブジェクトを生成する
5. **計画してから実行** — コードを書く前に複雑な機能の計画を立てる

## エージェントオーケストレーション

ユーザーの指示がなくともエージェントを能動的に活用すること:
- 複雑な機能要求 → **planner**
- コードを作成・修正した直後 → **code-reviewer**
- バグ修正・新機能 → **tdd-guide**
- アーキテクチャ判断 → **architect**
- セキュリティが関わるコード → **security-reviewer**
- 自律ループ・ループ監視 → **loop-operator**
- ハーネス設定の信頼性・コスト → **harness-optimizer**

独立したオペレーションには並列実行を用いる。複数エージェントを同時に起動すること。

## セキュリティガイドライン

**コミット前に必ず確認:**
- ハードコードされたシークレット(API キー、パスワード、トークン)が無いこと
- すべてのユーザー入力が検証されていること
- SQL インジェクション対策(パラメータ化クエリ)
- XSS 対策(HTML のサニタイズ)
- CSRF 保護が有効
- 認証・認可が検証済み
- 全エンドポイントにレートリミット
- エラーメッセージが機密情報を漏らさない

**シークレット管理:** シークレットを絶対にハードコードしない。環境変数またはシークレットマネージャを使用する。起動時に必須シークレットを検証する。露出したシークレットは直ちにローテートする。

**セキュリティ問題を発見した場合:** STOP → security-reviewer エージェントを使用 → CRITICAL 問題を修正 → 露出したシークレットをローテート → コードベース内に類似問題が無いかレビュー。

## コーディングスタイル

**イミュータビリティ(CRITICAL):** 既存オブジェクトを変更せず、常に新しいオブジェクトを生成する。変更を適用した新しいコピーを返す。

**ファイル構成:** 大きなファイル数個より、小さなファイルを多く用意する。標準は 200〜400 行、最大 800 行。種類別ではなく、機能・ドメイン別に整理する。凝集度を高く、結合度を低く保つ。

**エラーハンドリング:** 全レイヤでエラーを処理する。UI コードではユーザー向けに分かりやすいメッセージを提供する。サーバーサイドで詳細なコンテキストをログ出力する。エラーを黙って握りつぶさない。

**入力検証:** システム境界ですべてのユーザー入力を検証する。スキーマベースの検証を用いる。明確なメッセージとともに早期に失敗する。外部データを信用しない。

**コード品質チェックリスト:**
- 関数は小さく(<50 行)、ファイルは焦点を絞る(<800 行)
- ネストを深くしない(>4 階層)
- 適切なエラーハンドリング、ハードコードされた値が無い
- 読みやすく、適切に命名された識別子

## テスト要件

**最低カバレッジ: 80%**

テスト種別(すべて必須):
1. **ユニットテスト** — 個々の関数、ユーティリティ、コンポーネント
2. **インテグレーションテスト** — API エンドポイント、データベース操作
3. **E2E テスト** — クリティカルなユーザーフロー

**TDD ワークフロー(必須):**
1. 最初にテストを書く(RED) — テストは FAIL するはず
2. 最小限の実装を書く(GREEN) — テストが PASS するはず
3. リファクタ(IMPROVE) — カバレッジ 80% 以上を確認

失敗の原因究明: テストの分離 → モックの確認 → 実装の修正(テストが誤っている場合を除き、テストは修正しない)。

## 開発ワークフロー

1. **計画** — planner エージェントを使用し、依存関係とリスクを把握し、フェーズに分割する
2. **TDD** — tdd-guide エージェントを使用し、テストを先に書いて実装、リファクタする
3. **レビュー** — 直ちに code-reviewer エージェントを使用し、CRITICAL/HIGH の問題に対応する
4. **適切な場所に知識を残す**
   - 個人のデバッグメモ、好み、一時的なコンテキスト → 自動メモリ
   - チーム・プロジェクトの知識(アーキテクチャ決定、API 変更、運用手順) → プロジェクトの既存ドキュメント構造
   - 現在のタスクが該当ドキュメントやコードコメントを生成する場合は、同じ情報を他所に重複させない
   - 明確なドキュメント配置場所が無い場合は、新しいトップレベルファイルを作成する前に確認する
5. **コミット** — Conventional Commits 形式、包括的な PR サマリを記載する

## ワークフローサーフェスポリシー

- `skills/` が正式なワークフローサーフェスである。
- 新しいワークフローはまず `skills/` に追加する。
- `commands/` はレガシーなスラッシュエントリ互換のためのサーフェスであり、移行やハーネス間互換のために shim が必要な場合のみ追加・更新する。

## Git ワークフロー

**コミット形式:** `<type>: <description>` — 種別: feat, fix, refactor, docs, test, chore, perf, ci

**PR ワークフロー:** 全コミット履歴を分析 → 包括的なサマリを作成 → テスト計画を含める → `-u` フラグ付きでプッシュ。

## アーキテクチャパターン

**API レスポンス形式:** 成功フラグ、データペイロード、エラーメッセージ、ページネーションメタデータを含む一貫したエンベロープ。

**リポジトリパターン:** データアクセスを標準インターフェース(findAll, findById, create, update, delete)の背後にカプセル化する。ビジネスロジックは具体的なストレージ機構ではなく、抽象インターフェースに依存させる。

**スケルトンプロジェクト:** 実績のあるテンプレートを検索し、並列エージェント(セキュリティ、拡張性、関連性)で評価し、最適なものをクローンし、実証済み構造の中で反復する。

## パフォーマンス

**コンテキスト管理:** 大規模リファクタリングや複数ファイル機能では、コンテキストウィンドウの最後の 20% を避ける。感度の低いタスク(単発編集、ドキュメント、簡単な修正)はより高い使用率に耐えられる。

**ビルドのトラブルシューティング:** build-error-resolver エージェントを使用 → エラーを分析 → 段階的に修正 → 修正のたびに検証する。

## プロジェクト構成

```
agents/          — 60 specialized subagents
skills/          — 232 workflow skills and domain knowledge
commands/        — 75 slash commands
hooks/           — Trigger-based automations
rules/           — Always-follow guidelines (common + per-language)
scripts/         — Cross-platform Node.js utilities
mcp-configs/     — 14 MCP server configurations
tests/           — Test suite
```

`commands/` は互換のためリポジトリに残るが、長期的にはスキルファーストの方向に進む。

## 成功指標

- すべてのテストが 80% 以上のカバレッジで成功
- セキュリティ脆弱性が無い
- コードが読みやすく保守可能
- パフォーマンスが許容範囲内
- ユーザー要件を満たしている
