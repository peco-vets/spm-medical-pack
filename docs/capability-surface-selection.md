# Capability Surface 選択

ある能力がルール、スキル、MCP サーバー、またはプレーン CLI/API ワークフローに属するかを判断する際のルーティングガイドとして使う。

ECC はこれらサーフェスを交換可能とは扱わない。目標は、正確性を保持し、トークンコストを制御下に保ち、不必要なランタイムやサプライチェーン抗力を生み出さない最も狭いサーフェスに各能力を置くことである。

## 短いバージョン

- `rules/` はパスやイベントが一致したときに注入されるべき決定論的、常時オン制約用である。
- `skills/` はオンデマンドワークフロー、より豊かなプレイブック、関連時にのみロードされるべきトークン高価なガイダンス用である。
- `MCP` はセッションやクライアント全体で長期ツール/リソースサーフェスから恩恵を受ける対話型構造化能力用である。
- ローカル `CLI` またはリポジトリスクリプトは、永続的サーバーを必要としない単純な決定論的アクション用である。
- スキル内の直接 `API` コールは、完全な MCP サーバーが問題より重くなる狭いリモートアクション用である。

## 決定順序

これらの質問を順に問う:

1. これはパスやイベントが一致するたびに、モデル判断無しで起こるべきか?
   - `rule` を使う。
2. これは主にプレイブック、ワークフロー、またはタスクが実際に必要とするときのみロードされるべきアドバイザリレイヤか?
   - `skill` を使う。
3. 能力は複数のハーネスやクライアントが繰り返し呼ぶべき構造化された対話型ツール/リソースインターフェースを必要とするか?
   - `MCP` を使う。
4. これはサーバーを生かしておく必要無くスクリプトとして実行できる単純なローカルアクションか?
   - ローカル `CLI` エントリポイントまたはリポジトリスクリプトを使い、必要ならスキルで wrap する。
5. これはより大きなワークフロー内の 1 つの狭いリモート統合ステップだけか?
   - スキルまたはスクリプトから直接外部 `API` を呼ぶ。

## サーフェスごとのガイダンス

### Rules

ルールを使う対象:

- パススコープのコーディング不変条件
- 安全フロアと権限制約
- 常に適用されるべきハーネス/ランタイム制約
- モデル裁量に依存すべきでない決定論的リマインダ

ルールを使わない対象:

- 一致する各編集を肥大化させる大きなプレイブック
- オプショナルワークフロー
- 時々しか関係しない高価なドメインコンテキスト

### Skills

スキルを使う対象:

- マルチステップワークフロー
- 判断重視ガイダンス
- オンデマンドでのみロードするのに十分高価なドメインプレイブック
- スクリプト、API、MCP ツール、隣接スキル全体のオーケストレーション

決定論的ルーティングを真に欲しい静的不変条件のダンピンググラウンドとしてスキルを使わない。

### MCP

MCP を使うのは能力が以下から恩恵を受けるとき:

- 構造化ツール入出力
- 再利用可能リソースまたはプロンプト
- 繰り返しのクロスクライアント利用
- Claude Code、Codex、Cursor、OpenCode、関連ハーネス全体で動作すべき安定インターフェース
- 運用オーバーヘッドに見合う長期サーバープロセス

MCP を避けるとき:

- 作業がワンショットローカルコマンド
- サーバーがするのが 1 回 shell out するだけ
- サーバーがプロダクト価値以上にインストール/ランタイム負担を追加する

### CLI / リポジトリスクリプト

以下の場合はローカルスクリプトまたは CLI を優先:

- アクションが決定論的
- 起動が安価
- ワークフローが主にローカル
- 永続ツール/リソースサーフェスを公開する利点が無い

これは多くの場合以下に正しい選択:

- lint/test/build ラッパー
- ローカル変換
- 小さなインストーラ
- 起動ごとに 1 回実行されるコンテンツ生成

### 直接 API コール

以下のとき既存スキルまたはスクリプト内の直接 API コールを優先:

- 統合が狭い
- リモートアクションがより大きなワークフローの一部
- まだ再利用可能トランスポートサーフェスを必要としない

同じリモート統合が中心的、繰り返し、マルチクライアントになると、それを MCP サーフェスに昇格するシグナルとなる。

## コストと信頼性バイアス

2 つの選択肢がともに実現可能なとき:

- より小さなランタイムサーフェスを優先
- より低いトークンオーバーヘッドを優先
- より少ない外部可動部品のパスを優先
- 別のサードパーティ依存を導入するより ECC ネイティブパッケージングを優先

能力が明らかに保守、セキュリティ、インストール負担に値する場合を除き、外部プラグインやパッケージ依存をファーストクラス ECC サーフェスとして正常化しない。

## リポジトリポリシー

外部リポジトリからアイデアを持ち込む際:

- 外部依存ではなく根底にあるアイデアをコピーする
- ECC ネイティブルール、スキル、スクリプト、または MCP サーフェスとして再パッケージする
- ECC 用に機能が実質的に拡張または再形成されたら改名する
- 依存が意図的、監査済み、ワークフローの中心でない限り、無関係のサードパーティパッケージのインストールをユーザーに要求する指示を出荷しない

## 例

- `api/**` 編集に常に適用されるべきバックエンド auth 不変条件:
  - `rule`
- より深い API 設計とページネーションプレイブック:
  - `skill`
- 複数ハーネス全体で使われる再利用可能リモート検索サーフェス:
  - `MCP`
- ローカルファイルを読みレポートを書くワンショットリポジトリアナライザ:
  - ローカル `CLI` またはスクリプト、オプションで `skill` でラップ
- より広いカスタマオペワークフロー内の単一 billing ポータルセッション作成ステップ:
  - ワークフロー内の直接 `API` コール

## 実用ヒューリスティクス

不明な場合、より小さく始める:

- 決定論的不変条件には `rule` から始める
- ガイダンス/ワークフローには `skill` から始める
- ワンショット実行にはスクリプトから始める
- 構造化サーバー境界が明らかに自身を支払うときのみ `MCP` に昇格する
