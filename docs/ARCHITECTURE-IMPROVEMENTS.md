# アーキテクチャ改善提案

本ドキュメントは、Everything Claude Code (ECC) プロジェクトのアーキテクトレベル改善を捕捉する。保守性、一貫性、長期的な品質改善を狙う Claude Code コーディングアーキテクトの視点で書かれている。

---

## 1. ドキュメントと信頼の単一情報源

### 1.1 Agent/Command/Skill カウントの同期

**問題:** AGENTS.md は "13 specialized agents, 50+ skills, 33 commands" と記載しているが、リポジトリには **16 agents**、**65+ skills**、**40 commands** が存在する。README やその他ドキュメントも数字が一致しない。これはコントリビューターとユーザーに混乱をもたらす。

**推奨:**

- **信頼の単一情報源:** カウント (および任意でテーブル) をファイルシステムまたは小さなマニフェストから導出する。選択肢:
  - **Option A:** スクリプト (例: `scripts/ci/catalog.js`) を追加し、`agents/*.md`、`commands/*.md`、`skills/*/SKILL.md` をスキャンして JSON/Markdown を出力する。CI とドキュメントがこれを消費する。
  - **Option B:** 1 つの `docs/catalog.json` (または YAML) を維持し、エージェント、コマンド、スキルをメタデータ付きでリストする。スクリプトとドキュメントはここから読む。追加・削除時に更新する規律が必要。
- **短期:** AGENTS.md、README.md、CLAUDE.md を実際のカウントで手動同期し、新エージェント (例: chief-of-staff、loop-operator、harness-optimizer) をエージェントテーブルに記載する。

**影響:** High — 第一印象とコントリビューターの信頼に影響する。

---

### 1.2 Command → Agent/Skill マップ

**問題:** 「どのコマンドがどのエージェントやスキルを使うか」の機械可読・人間可読な単一のマップが無い。これは README テーブルや個別コマンドの `.md` ファイルに散在しており、ドリフトしうる。

**推奨:**

- **コマンドレジストリ** (例: `docs/` または command ファイルの frontmatter) を追加し、各コマンドについて name、description、主要エージェント、参照スキルをリストする。コマンドファイル内容から生成するか、手動でメンテナンスできる。
- ドキュメント (例: `docs/COMMAND-AGENT-MAP.md`) または生成されたカタログに「マップ」を公開し、発見可能性とツーリング (例: "どのコマンドが tdd-guide を使うか?") に役立てる。

**影響:** Medium — 発見可能性とリファクタリングの安全性を改善する。

## 2. テストと品質

### 2.1 テスト発見 vs ハードコードリスト

**問題:** `tests/run-all.js` は **ハードコードされたリスト** のテストファイルを使う。新しいテストファイルは誰かが `run-all.js` を更新しない限り走らない。脱漏でカバレッジが不完全になりうる。

**推奨:**

- **glob ベースの発見:** パターン (例: `tests/` 配下の `**/*.test.js`) でテストファイルを発見し実行する。特殊ケースのオプションの allowlist/denylist 付きで。これにより新テストが自動的にスイートの一部になる。
- 単一エントリポイント (`tests/run-all.js`) を保ち、発見されたテストを実行して結果を集約する。

**影響:** High — 新テストが存在するが実行されない回帰を防ぐ。

---

### 2.2 テストカバレッジメトリクス

**問題:** カバレッジツール (例: nyc/c8/istanbul) が無い。プロジェクトは自身のスクリプトに対して「80% 以上のカバレッジ」を主張できず、カバレッジは暗黙のままである。

**推奨:**

- Node スクリプト用カバレッジツール (例: `c8` または `nyc`) を導入し、CI で実行する。ベースライン (例: 60%) から始め、徐々に上げる。あるいは少なくとも CI で失敗させずにカバレッジを報告し、チームがトレンドを見られるようにする。
- `scripts/` (lib + hooks + ci) を主要ターゲットとし、必要なら 1 回限りのスクリプトは除外する。

**影響:** Medium — AGENTS.md ガイダンス (80% 以上のカバレッジ) と整合し、未テストのパスを浮上させる。

---

## 3. スキーマと検証

### 3.1 CI で Hooks JSON Schema を使う

**問題:** `schemas/hooks.schema.json` は存在し、フック設定形状を定義しているが、`scripts/ci/validate-hooks.js` は **これを使わない**。検証 (VALID_EVENTS、構造) が重複し、スキーマからドリフトしうる。

**推奨:**

- `validate-hooks.js` で JSON Schema バリデータ (例: `ajv`) を使い、`hooks/hooks.json` を `schemas/hooks.schema.json` に対して検証する。バリデータを構造の単一情報源として保持し、スクリプトにはフック固有チェック (例: インライン JS 文法) のみを残す。
- スキーマとバリデータが同期され続け、hooks.json 内の `$schema` 経由で IDE/エディタ検証が可能になる。

**影響:** Medium — ドリフトを減らし、フック編集時のコントリビューター体験を改善する。

---

## 4. クロスハーネスと i18n

### 4.1 Skill/Agent サブセット同期 (.agents/skills, .cursor/skills)

**問題:** `.agents/skills/` (Codex) と `.cursor/skills/` は `skills/` のサブセットである。メインリポジトリでスキルを追加・削除する際、これらサブセットを手動で更新する必要があり、忘れられうる。

**推奨:**

- CONTRIBUTING.md に、スキル追加時に `.agents/skills` と `.cursor/skills` の更新が必要な場合があること(と更新方法)を文書化する。
- 任意: `skills/` をサブセットと比較する CI チェックまたはスクリプトを追加し、規約や小さなマニフェストに従ってスキルが片側にしか無い場合に失敗または警告する。

**影響:** Low-Medium — クロスハーネスドリフトを減らす。

---

### 4.2 翻訳ドリフト (docs/ zh-CN, zh-TW, ja-JP)

**問題:** `docs/` 内の翻訳はエージェント、コマンド、スキルを重複させている。英語ソースが進化するにつれ、翻訳は明確なプロセスやツーリング無しに古くなりうる。

**推奨:**

- **翻訳プロセス** を文書化する: いつ更新するか (例: リリース時)、各ロケールを誰が所有するか、陳腐な内容を検出する方法 (例: diff ファイルリストや主要セクション)。
- 検討事項: 翻訳ステータスファイル (例: `docs/i18n-status.md`) または翻訳ファイルの存在/タイムスタンプをチェックし、翻訳より英語の方が新しく更新された場合に警告する CI。
- 長期: 翻訳が英語ソースと同じ構造を参照するよう、抽出/プレースホルダ形式 (例: i18n キー) を検討する。

**影響:** Medium — 非英語ユーザーの体験を改善し、陳腐な翻訳による混乱を減らす。

---

## 5. フックとスクリプト

### 5.1 フックランタイムの一貫性

**問題:** フックは一貫した Node モードディスパッチサーフェスを維持すべきである。継続学習観測は現在 `run-with-flags.js` と `observe-runner.js` を介してディスパッチされ、これは shell-mode フックエントリを公開せずに既存の `observe.sh` 実装に委任する。

**推奨:**

- 可能なら新しいフックには Node を優先する (クロスプラットフォーム、単一ランタイム)。shell が必要な場合は理由を文書化し、サーフェスを小さく保つ。
- すべてのコードパス (shell を含む) で `ECC_HOOK_PROFILE` と `ECC_DISABLED_HOOKS` が尊重されることを保証し、挙動の一貫性を保つ。

**影響:** Low — 現在の設計を維持する。より多くのフックが Node に移行すれば改善する。

## 6. サマリテーブル

| 領域              | 改善                                  | 優先度   | 工数     |
|-------------------|---------------------------------------|----------|----------|
| Doc sync          | AGENTS.md/README のカウント・テーブル同期 | High     | Low      |
| Single source     | カタログスクリプトまたはマニフェスト   | High     | Medium   |
| Test discovery    | glob ベースのテストランナー            | High     | Low      |
| Coverage          | c8/nyc 追加と CI カバレッジ            | Medium   | Medium   |
| Hook schema in CI | スキーマ経由で hooks.json を検証       | Medium   | Low      |
| Command map       | Command → agent/skill レジストリ       | Medium   | Medium   |
| Subset sync       | .agents/.cursor のドキュメント・CI     | Low-Med  | Low-Med  |
| Translations      | プロセス + 陳腐検出                    | Medium   | Medium   |
| Hook runtime      | Node を優先、shell 利用を文書化         | Low      | Low      |

## 7. クイックウィン (即時)

1. **AGENTS.md 更新:** エージェントカウントを 16 にし、chief-of-staff、loop-operator、harness-optimizer をエージェントテーブルに追加。スキル/コマンドカウントをリポジトリと整合させる。
2. **テスト発見:** `run-all.js` を `tests/` 配下の `**/*.test.js` を発見するよう変更 (オプション allowlist 付き)。これにより新テストが常に走るようになる。
3. **フックスキーマ配線:** `validate-hooks.js` で ajv (または類似) を用いて `hooks/hooks.json` を `schemas/hooks.schema.json` に対して検証し、フック固有チェックのみをスクリプトに残す。

これら 3 つは 1〜2 セッションで完了でき、一貫性と信頼性を実質的に向上させる。
