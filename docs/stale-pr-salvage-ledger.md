# 陳腐 PR サルベージ台帳

本台帳は陳腐、コンフリクト、またはクローズされた PR から回収された有用な作業を記録する。
ルールはシンプル: キュークリーンアップは陳腐 PR をクローズするが、有用な作業を破棄しない。メンテナはクローズされた diff を検査し、互換ピースを新ブランチに port し、ソース PR をクレジットすべき。

## 分類状態

| 状態 | 意味 |
| --- | --- |
| Salvaged | 有用な作業がメンテナ PR を通じて現 `main` に port された。 |
| Already present | 現 `main` がサルベージ前に既に有用な作業を含んでいた。 |
| Superseded | 現 `main` が同じ問題を異なる方法で解決した。 |
| Skipped | PR が偶発的、広すぎる、安全でない、または port するには低シグナルすぎた。 |
| 翻訳者/手動レビュー | コンテンツは有用かもしれないが、インポート前に人間言語/ドメインレビューが必要。 |

## 残る手動レビューバックログ

残るおそらく有用なバックログは、言語オーナーレビュー無しに自動 port するには安全でない翻訳/ローカライズ作業である。このテールは Linear ITO-55 にアタッチされており、リリースブロッキングサルベージタスクではない。リリース作業はバックログが記録されたまま盲目的インポートから除外されていることを検証するだけでよい:

- #1687 zh-CN ローカライズテール
- #1609 Persian README 翻訳
- #1563 zh-TW README 同期
- #1564 Turkish README 同期
- #1565 pt-BR README 同期

ハンドリングルール:

1. これら PR を翻訳者/手動レビューに保つ。
2. 将来の作業をサーフェスで分割する: エージェント、コマンド、トップレベルドキュメント、リリースとカウントサーフェス、その後スキル。
3. 古いバージョンやカタログカウント事実を運ぶ陳腐なトップレベルドキュメントをインポートしない。
4. 元の作者が現リベースで戻らない限り古い PR を再オープンしない。メンテナ側サルベージは帰属付きで新ブランチで起こるべき。

## 将来のクリーンアップルール

すべての陳腐/コンフリクト PR クリーンアップバッチで:

1. キューポリシーに基づいて PR をクローズまたはコメントする。
2. ソース PR を本台帳または日付付き後継台帳に追加する。
3. salvaged、already present、superseded、skipped、または翻訳者/手動レビューとして分類する。
4. 有用なら、新メンテナブランチで小さな互換スライスを port する。
5. メンテナ PR 本文でソース PR と作者をクレジットする。
