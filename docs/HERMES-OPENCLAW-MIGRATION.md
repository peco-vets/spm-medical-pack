# Hermes / OpenClaw → ECC マイグレーション

本ドキュメントは Hermes または OpenClaw スタイルのオペレータセットアップを現行 ECC モデルに移行する公開マイグレーションガイドである。

目標はプライベートなオペレータワークスペースをバイト単位で再現することではない。

目標は有用なワークフローサーフェスを保持することである:

- 再利用可能スキル
- 安定した自動化エントリポイント
- クロスハーネス可搬性
- スケジューラ / リマインダ / ディスパッチ
- 永続的コンテキストとオペレータメモリ

一方、プライベートに留めるべき部分は除去する:

- シークレット
- 個人データセット
- アカウントトークン
- ローカル専用ビジネスアーティファクト

## マイグレーションテーゼ

Hermes と OpenClaw を最終ランタイムではなく、ソースシステムとして扱う。

ECC は永続的な公開システムである:

- スキル
- エージェント
- コマンド
- フック
- インストールサーフェス
- セッションアダプタ
- ECC 2.0 コントロールプレーン作業

Hermes と OpenClaw は、ECC ネイティブサーフェスに蒸留できる繰り返しオペレータワークフローを含むため、有用な入力である。

つまり最短で安全なパスは:

1. 再利用可能な挙動を抽出する
2. それを ECC ネイティブのスキル、フック、ドキュメント、またはアダプタ作業に翻訳する
3. シークレットと個人データはリポジトリ外に保つ

## 現在のワークスペースモデル

現在のワークスペース分割を一貫して使う:

- ライブコード作業は `~/GitHub` 配下のクローン済みリポジトリで行う
- リポジトリ固有のアクティブ実行コンテキストは、リポジトリレベルの `WORKING-CONTEXT.md` に存在する
- より広範な非コードコンテキストは KB/アーカイブレイヤに存在しうる
- 永続的クロスマシン真実は GitHub、Linear、ナレッジベースを優先すべき

公開リポジトリ内にシャドウプライベートワークスペースを再構築しないこと。

## 翻訳マップ

### 1. スケジューラ / cron レイヤ

ソース例:

- `cron/scheduler.py`
- `jobs.py`
- 繰り返しレディネスまたは説明責任ループ

翻訳先:

- 利用可能なら Claude ネイティブスケジューリング
- ローカル再現性のための ECC フック / コマンド自動化
- Issue `#1050` 配下の ECC 2.0 スケジューラ作業

今日、リポジトリは既に正しい公開フレーミングを持つ:

- 低レイテンシリポジトリローカル自動化のためのフック
- 明示的オペレータアクションのためのコマンド
- 将来の長期スケジューリング/コントロールプレーンとしての ECC 2.0

### 2. ゲートウェイ / ディスパッチレイヤ

ソース例:

- Hermes ゲートウェイ
- モバイルディスパッチ / リモートナッジ
- アクティブセッション間のオペレータルーティング

翻訳先:

- ECC セッションアダプタとコントロールプレーン作業
- オーケストレーション/セッション検査コマンド
- 以下配下の ECC 2.0 コントロールプレーンバックログ:
  - `#1045`
  - `#1046`
  - `#1047`
  - `#1048`

公開リポジトリはアダプタ境界とコントロールプレーンモデルを記述すべきで、リモートオペレータシェルが既に完全に GA であるかのように見せかけてはならない。

### 3. メモリレイヤ

ソース例:

- `memory_tool.py`
- ローカルオペレータメモリ
- ビジネス / 運用コンテキストストア

翻訳先:

- `knowledge-ops`
- リポジトリ `WORKING-CONTEXT.md`
- GitHub / Linear / KB バック永続コンテキスト
- `#1049` 配下の将来の深いメモリ作業

重要な区別は:

- リポジトリ実行コンテキストはリポジトリ近くに属する
- より広範な非コードメモリは KB/アーカイブシステムに属する
- 公開リポジトリは境界を文書化すべきで、プライベートメモリダンプを保存すべきでない

### 4. スキルレイヤ

ソース例:

- Hermes スキル
- OpenClaw スキル
- 生成されたオペレータプレイブック

翻訳先:

- ワークフローが再利用可能なら ECC ネイティブトップレベルスキル
- 内容がテンプレートのみなら docs/examples
- 挙動が知識形よりも手続き的ならフックまたはコマンド

最近この方法でサルベージされた例:

- `knowledge-ops`
- `github-ops`
- `hookify-rules`
- `automation-audit-ops`
- `email-ops`
- `finance-billing-ops`
- `messages-ops`
- `research-ops`
- `terminal-ops`
- `ecc-tools-cost-audit`

### 5. ツール / サービスレイヤ

ソース例:

- カスタムサービスラッパー
- API キーバックローカルツール
- ブラウザ自動化グルー

翻訳先:

- コネクタが存在するなら MCP バックサーフェス
- ワークフローロジックが真の資産なら ECC ネイティブオペレータスキル
- 欠けている部分がセッション/ランタイム調整ならアダプタ/コントロールプレーン作業

プライベートワークフローが依存していたという理由だけで、不透明なサードパーティランタイムを ECC にインポートしないこと。

ワークフローが価値ある場合:

1. 挙動を理解する
2. 最小 ECC ネイティブバージョンを再構築する
3. ローカルに必要な認証/コネクタを文書化する

## 既に公開されているもの

現リポジトリは既にマイグレーションの意味のある部分をカバーしている:

- ECC 2.0 アダプタ/コントロールプレーンディスカバリドキュメント
- オーケストレーション/セッション検査基盤
- オペレータワークフロースキル
- コスト / billing / ワークフロー監査スキル
- クロスハーネスインストールサーフェス
- 設定とエージェントサーフェススキャン用 AgentShield

つまりマイグレーション問題はもはや「ゼロから始める」ではない。

主に:

- 欠けているプライベートワークフローを蒸留する
- 公開ドキュメントを明確化する
- ECC 2.0 オペレータ/コントロールプレーン構築を継続する

ECC 2.0 は現在、境界付きマイグレーション監査エントリポイントを出荷する:

- `ecc migrate audit --source ~/.hermes`
- `ecc migrate plan --source ~/.hermes --output migration-plan.md`
- `ecc migrate scaffold --source ~/.hermes --output-dir migration-artifacts`
- `ecc migrate import-skills --source ~/.hermes --output-dir migration-artifacts/skills`
- `ecc migrate import-tools --source ~/.hermes --output-dir migration-artifacts/tools`
- `ecc migrate import-plugins --source ~/.hermes --output-dir migration-artifacts/plugins`
- `ecc migrate import-schedules --source ~/.hermes --dry-run`
- `ecc migrate import-remote --source ~/.hermes --dry-run`
- `ecc migrate import-env --source ~/.hermes --dry-run`
- `ecc migrate import-memory --source ~/.hermes`

これを最初に使い、レガシーワークスペースをインベントリして、検出されたサーフェスを現行 ECC2 スケジューラ、リモートディスパッチ、メモリグラフ、テンプレート、手動翻訳レーンにマップする。

## 推奨ブリングアップ順序

1. 公開 ECC リポジトリを canonical な再利用可能レイヤとして保つ。
2. 再利用可能 Hermes/OpenClaw ワークフローを 1 レーンずつ ECC ネイティブスキルに port する。
3. プライベート認証と個人コンテキストはリポジトリ外に保つ。
4. 永続真実として GitHub / Linear / KB システムを使う。
5. ECC 2.0 を完成したプロダクトとしてではなく、ネイティブオペレータシェルへのパスとして扱う。

## 決定ルール

Hermes または OpenClaw アーティファクトをレビューする際、以下を問う:

1. これはオペレータ全体で再利用可能か、それとも個人専用か?
2. アセットは主に知識、手続き、ランタイム挙動のどれか?
3. 以下のどれになるべきか:
   - スキル
   - コマンド
   - フック
   - ドキュメント/例
   - コントロールプレーン issue
4. 公開出荷するとシークレット、プライベートデータセット、または個人運用状態を漏らすか?

再利用可能なサーフェスのみを出荷する。
