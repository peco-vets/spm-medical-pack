# スキル適応ポリシー

ECC は外部リポジトリからのアイデアを受理するが、出荷されるスキルは ECC ネイティブサーフェスになる必要がある。

## デフォルトルール

別のオープンソースリポジトリ、プロンプトパック、プラグイン、ハーネス、または個人設定からコントリビューションが始まるとき:

- 根底にあるアイデア、ワークフロー、または構造をコピー
- ECC の現インストールサーフェス、検証フロー、リポジトリ規約に適応
- 不必要な外部ブランディング、依存仮定、アップストリーム固有フレーミングを削除

目標は、ECC を他人のランタイム周辺の薄いラッパーにせずに再利用することである。

## 元の名前を保つとき

以下のすべてが真の場合のみ元のスキル名を保つ:

- コントリビューションが直接 port に近い
- 名前が既に説明的で中立的
- サーフェスが依然アップストリームコンセプトのように振る舞う
- リポジトリ内に既により良い ECC ネイティブ名が無い

例:

- `nestjs-patterns` のようなフレームワーク名
- ベンダーピッチではなく主題であるプロトコルやプロダクト名

## 改名するとき

ECC が元の作業を意味のあるほど拡張、絞り込み、または再パッケージしたときスキルを改名する。

典型的トリガ:

- ECC が実質的な新挙動、構造、ガイダンスを追加
- 元の名前がワークフローフォワードではなくベンダーフォワードまたはコミュニティブランドフォワード
- コントリビューションが既存 ECC サーフェスと重複しより明確な境界を必要とする
- コントリビューションが現在、リテラル port ではなく能力、オペレータワークフロー、またはポリシーレイヤとして適合する

例:

- 再利用可能グラフプリミティブを `social-graph-ranker` として保ち、より広いワークフローレイヤを `lead-intelligence` や `connections-optimizer` にする
- スコープが実質的に変わった場合、曖昧なインポート計画ラベルより `product-capability` のような ECC ネイティブ名を優先する

## 依存ポリシー

ECC は仕事を片付ける最も狭いネイティブサーフェスを優先する:

- 決定論的制約には `rules/`
- オンデマンドワークフローには `skills/`
- 長期対話型ツール境界が正当化されるとき MCP
- 決定論的ワンショット実行にはローカルスクリプト/CLI
- リモートコールが狭く MCP を正当化しないときは直接 API

主に未検証のサードパーティパッケージをインストールまたは信頼するようユーザーに伝えるために存在するスキルを出荷しない。

外部機能が保つ価値があるなら:

- 実用的なら ECC 内で関連ロジックをベンダリングまたは再作成する
- または統合をオプショナルに保ち外部として明確にマークする
- 明示的正当化無しに新外部依存をデフォルトパスにしない

## レビュー質問

コントリビュートされたスキルをマージする前に、これらに答える:

1. これは ECC で真に再利用可能なサーフェスか、それとも別ツールのドキュメントだけか?
2. 現名前が依然 ECC 形状のサーフェスに合致するか?
3. この挙動の大部分を既に所有する ECC スキルが既にあるか?
4. コンセプトをインポートしているか、それとも他人のプロダクトアイデンティティをインポートしているか?
5. ECC ユーザーがアップストリームリポジトリを知らずに本スキルの目的を理解できるか?

それら回答が弱いなら、もっと適応するか、スコープを絞るか、出荷しない。
