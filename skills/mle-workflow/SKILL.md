# Machine Learning Engineering ワークフロー

モデル作業を、明確なデータ契約、再現可能なトレーニング、測定可能な品質ゲート、デプロイ可能なアーティファクト、運用モニタリングを備えた本番 ML システムに変換するためにこのスキルを使う。

## 起動するタイミング

- 本番 ML フィーチャー、モデルリフレッシュ、ランキングシステム、レコメンダー、分類器、エンベディングワークフロー、または予測パイプラインの計画またはレビュー
- ノートブックコードを再利用可能なトレーニング、評価、バッチ推論、またはオンライン推論パイプラインに変換
- モデル昇格基準、オフライン/オンライン評価、実験追跡、またはロールバックパスの設計
- データドリフト、ラベルリーク、古いフィーチャー、アーティファクト不一致、または訓練/サービングロジックの不整合による失敗のデバッグ
- モデルモニタリング、カナリアロールアウト、シャドートラフィック、またはデプロイ後品質チェックの追加

## スコープキャリブレーション

目の前のシステムに合うレーンのみ使う。このスキルはランキング、検索、レコメンデーション、分類器、予測、エンベディング、LLM ワークフロー、異常検知、バッチ分析に有用だが、すべてに 1 つのアーキテクチャを強制すべきではない。

- すべてのモデルが教師あり、ラベル、オンラインサービング、フィーチャーストア、PyTorch、GPU、ヒューマンレビュー、A/B テスト、またはリアルタイムフィードバックを持つと仮定しない
- データ契約、ベースライン、評価スクリプト、ロールバックノートが変更をレビュー可能にするときに、重量級 MLOps 機構を追加しない
- プロジェクトにラベル、遅延結果、スライス定義、本番トラフィック、またはモニタリング所有権が欠けている場合は仮定を明示する
- 例を交換可能な足場として扱う。メトリクス、サービングモード、データストア、ロールアウト機構をプロジェクトネイティブの同等品に置き換える

## 関連スキル

- Python 実装と pytest カバレッジには `python-patterns` と `python-testing`
- ディープラーニングモデル、データローダー、デバイス処理、トレーニングループには `pytorch-patterns`
- 昇格ゲートとエージェント支援回帰チェックには `eval-harness` と `ai-regression-testing`
- データストレージと分析サーフェスには `database-migrations`、`postgres-patterns`、`clickhouse-io`
- サービング、シークレット、コンテナ、本番ハードニングには `deployment-patterns`、`docker-patterns`、`security-review`

## SWE サーフェスを再利用する

MLE をソフトウェアエンジニアリングから分離して扱わない。ほとんどの ECC SWE ワークフローは ML システムに直接適用される。多くの場合、より厳格な失敗モードで。

推奨される `minimal --with capability:machine-learning` インストールは、このスキルと並んでコアエージェントサーフェスを利用可能に保つ。スキルのみまたはエージェント限定ハーネスの場合、ターゲットがエージェントをサポートする場所では `skill:mle-workflow` と `agent:mle-reviewer` をペアにする。

| SWE サーフェス | MLE での使用 |
|-------------|---------|
| `product-capability` / `architecture-decision-records` | モデル作業を明示的な製品契約に変換し、不可逆的データ、モデル、ロールアウト選択を記録 |
| `repo-scan` / `codebase-onboarding` / `code-tour` | 並行 ML スタックを導入する前に既存のトレーニング、フィーチャー、サービング、評価、モニタリングパスを見つける |
| `plan` / `feature-dev` | データ、評価、サービング、ロールバックフェーズを持つ製品機能としてモデル変更をスコープ |
| `tdd-workflow` / `python-testing` | 実装前にフィーチャー変換、分割ロジック、メトリック計算、アーティファクトローディング、推論スキーマをテスト |
| `code-reviewer` / `mle-reviewer` | コード品質に加えて ML 固有のリーク、再現性、昇格、モニタリングリスクをレビュー |
| `build-fix` / `pr-test-analyzer` | 壊れた CI、フレーキーな評価、欠落フィクスチャ、環境固有のモデルや依存関係の失敗を診断 |
| `quality-gate` / `test-coverage` | 変換、メトリクス、推論契約、昇格ゲート、ロールバック挙動の自動証拠を要求 |
| `eval-harness` / `verification-loop` | オフラインメトリクス、スライスチェック、レイテンシバジェット、ロールバックドリルを反復可能なゲートに変換 |
| `ai-regression-testing` | すべての本番バグを回帰として保存: 欠落フィーチャー、古いラベル、不正アーティファクト、スキーマドリフト、またはサービング不一致 |
| `api-design` / `backend-patterns` | 予測 API、バッチジョブ、冪等な再トレーニングエンドポイント、レスポンスエンベロープを設計 |
| `database-migrations` / `postgres-patterns` / `clickhouse-io` | ラベル、フィーチャースナップショット、予測ログ、実験メトリクス、ドリフト分析をバージョン管理 |
| `deployment-patterns` / `docker-patterns` | ヘルスチェック、リソース制限、ロールバックを備えた再現可能なトレーニングとサービングイメージをパッケージ |
| `canary-watch` / `dashboard-builder` | モデルバージョン、スライス、ドリフト、レイテンシ、コスト、遅延ラベルダッシュボードでロールアウト健全性を可視化 |
| `security-review` / `security-scan` | モデルアーティファクト、ノートブック、プロンプト、データセット、ログのシークレット、PII、安全でないデシリアル化、サプライチェーンリスクをチェック |
| `e2e-testing` / `browser-qa` / `accessibility` | 説明可能性とフォールバック UI 状態を含む、予測を消費する重要な製品フローをテスト |
| `benchmark` / `performance-optimizer` | スループット、p95 レイテンシ、メモリ、GPU 使用率、予測または再トレーニングあたりのコストを測定 |
| `cost-aware-llm-pipeline` / `token-budget-advisor` | 最大モデルにデフォルト設定する代わりに、品質、レイテンシ、予算で LLM/エンベディングワークロードをルーティング |
| `documentation-lookup` / `search-first` | コーディング前にモデルサービング、フィーチャーストア、ベクトル DB、評価ツーリングの現在のライブラリ挙動を検証 |
| `git-workflow` / `github-ops` / `opensource-pipeline` | 明瞭なスコープ、生成アーティファクト除外、再現可能なテスト証拠で MLE 変更をレビュー用にパッケージ |
| `strategic-compact` / `dmux-workflows` | 長い ML 作業を並列トラックに分割: データ契約、評価ハーネス、サービングパス、モニタリング、ドキュメント |

これらのシミュレーションを MLE 作業の計画・レビュー時のカバレッジチェックとして使う。強い MLE ワークフローは、各タスクを明示的契約、再利用可能 SWE サーフェス、自動証拠、レビュー可能なアーティファクトに還元すべきである。

|----|-----------------|----------------------|-----------------|------------------------|

## Iteration Compact

モデルコードに触れる前に、作業を 1 つのレビュー可能なアーティファクトに圧縮する。これは PR 説明に収まるほど短く、別のエンジニアがトレードオフに挑戦できるほど精密であるべきである。

```text
Goal:
Who cares:
Decision owner:
User or system action changed by the model:
Success metric:
Guardrail metrics:
Mistake budget:
Unacceptable mistakes:
Acceptable mistakes:
Assumptions:
Constraints:
Labels and data snapshot:
Baseline:
Candidate signals:
Threshold or config plan:
Eval slices:
Known risks:
Next experiment:
Rollback or fallback:
```

このコンパクトは強い SWE 設計ノートの MLE 同等品である。チームが誰も信頼しないメトリックを最適化したり、実際のエラーモードに対処しないフィーチャーを追加したり、ロールバックなしで複雑さを出荷したりすることを防ぐ。

## 決定脳

タスクが曖昧、高インパクト、またはメトリック中心の場合、このループを使う:

1. モデルからではなく決定から始める。ダウンストリーム挙動を変えるアクションを命名する
2. 誰が気にするかと理由を命名する。異なるステークホルダーは false positive、false negative、レイテンシ、計算費用、不透明性、または機会損失に対して異なるコストを支払う
3. 曖昧さを仮説に変換する。どのシグナルが結果を分離するか、何が反証するか、どの単純なベースラインが打ち負かしにくいべきかを問う
4. カスタムシステムを発明する前に先行研究または近い既知の問題を調査する
5. `(probability, confidence) x (cost, severity, importance, impact)` で選択肢をスコアリングする
6. 敵対的挙動、インセンティブ、選択的開示、分布シフト、フィードバックループを考慮する
7. 最も重要なミスを減らす最も単純な変更を優先する。単純性は怠惰ではない。反復速度を維持しながらブランダーを最小化する方法である
8. 決定、証拠、反論、次の可逆ステップをキャプチャする

## メトリクスとミスのエコノミクス

習慣ではなく失敗コストからメトリクスを選ぶ:

- 抽象的な精度ではなく具体的な false positive と false negative について議論できるよう、混同行列を早期に使う
- 不正な肯定的決定のコストが支配的な場合は精度を優先する
- 見逃した肯定的のコストが支配的な場合は再現率を優先する
- 精度/再現率のトレードオフが真にバランスし説明可能な場合のみ F1 を使う
- 単一閾値より順序品質が重要な場合は AUC やランキングメトリクスを使う
- 実現可能なモデル複雑さを形成するため、レイテンシ、スループット、メモリ、コストをファーストクラスメトリクスとして追跡する
- オフライン獲得を祝う前に、ベースラインと現在の本番モデルと比較する
- 実世界のフィードバックシグナルをバイアス、ラグ、カバレッジギャップを伴う遅延ラベルとして扱う。分析なしにグラウンドトゥルースとして扱わない

すべてのメトリック選択は、どのミスを安くし、どのミスをより起こりやすくし、誰がそのコストを吸収するかを述べるべきである。

## データとフィーチャー仮説

フィーチャーは分離の理論から来るべきである:

- テキスト、カテゴリフィールド、数値履歴、グラフ関係、最近性、頻度、集計は候補シグナルファミリーであり、自動的なフィーチャーではない
- すべてのフィーチャーファミリーについて、なぜそれが結果を分離すべきか、どのように将来情報をリークし得るかを述べる
- ノイジーラベルには、裁定、ラベル信頼度、ソフトターゲット、または信頼度重み付けを考慮する
- クラス不均衡には、重み付き損失、リサンプリング、閾値移動、キャリブレーション済み決定ルールを比較する
- 欠損値については、不在が情報的か、補完可能か、または棄却の理由か決める
- 外れ値については、クリップ、バケット化、調査、または希少だが重要なシグナルとして保持するか決める
- 相関フィーチャーについては、それらが冗長か、不安定か、利用不可能な将来状態のプロキシかチェックする

エラー分析で、追加シグナルや容量が修正できるであろう理由でベースラインが失敗していると示すまで、モデル複雑さを追加しない。

## エラー分析ループ

各ベースライン、トレーニングラン、閾値変更、または設定変更後:

1. ミスを false positive、false negative、棄却、低信頼度ケース、システム失敗に分割する
2. 共有特性でエラーをクラスタリングする: 言語、エンティティタイプ、ソース、時間、地理、デバイス、スパース性、最近性、フィーチャー鮮度、ラベルソース、モデルバージョン
3. モデルミスをデータバグ、ラベル曖昧性、製品曖昧性、計装ギャップ、サービング不一致から分離する
4. 各主要クラスタを 4 つの動きの 1 つに辿る: より良いラベル、より良いフィーチャー、より良い閾値/設定、より良い製品フォールバック
5. すべての重要なミスを回帰テスト、評価スライス、ダッシュボードパネル、またはランブックエントリとして保存する
6. 次の反復を曖昧な「モデル改善」タスクではなく反証可能な実験として書く

最強の MLE ループは train -> metric -> ship ではない。mistake -> cluster -> hypothesis -> experiment -> evidence -> simpler system である。

## 観察台帳

コード、PR、実験レポート、またはランブックの傍にコンパクトな決定と証拠の軌跡を保持する:

```text
Iteration:
Change:
Why this mattered:
Metric movement:
Slice movement:
False positives:
False negatives:
Unexpected errors:
Decision:
Tradeoff accepted:
Lesson captured:
Regression added:
Debt created:
Next iteration:
```

台帳を使ってモデル作業を累積的にする。目標は各反復が次の決定を容易にすることであり、単に別のアーティファクトを生み出すことではない。

## コアワークフロー

### 1. 予測契約を定義する

モデルコードを書く前に製品レベル契約をキャプチャする:

- 予測ターゲットと決定オーナー
- 入力エンティティ、出力スキーマ、信頼度/キャリブレーションフィールド、許容レイテンシ
- バッチ、オンライン、ストリーミング、またはハイブリッドサービングモード
- モデル、フィーチャーストア、または依存関係が利用不可能なときのフォールバック挙動
- 高インパクト決定のためのヒューマンレビューまたはオーバーライドパス
- 入力、予測、ラベルのプライバシー、保持、監査要件

「モデルを改善する」を要件として受け入れない。モデルを観察可能な製品挙動と測定可能な受け入れゲートに結びつける。

### 2. データ契約をロックする

すべての ML タスクには明示的なデータ契約が必要である:

- エンティティ粒度と主キー
- ラベル定義、ラベルタイムスタンプ、ラベル可用性遅延
- フィーチャータイムスタンプ、鮮度 SLA、ポイントインタイム結合ルール
- トレーニング、検証、テスト、バックテスト分割ポリシー
- 必須カラム、許可される null、範囲、カテゴリ、単位
- トレーニングアーティファクトやログに入ってはいけない PII または機密フィールド
- 再現性のためのデータセットバージョンまたはスナップショット ID

まずリークから守る。予測時に利用不可能なフィーチャー、または将来情報を使って結合されるフィーチャーは、削除するか分析専用パスに移動する。

### 3. 再現可能なパイプラインを構築する

トレーニングコードは隠れたノートブック状態なしに別のエンジニアが実行可能であるべきである:

- すべてのハイパーパラメータとパスに型付き設定ファイルまたはデータクラスを使う
- パッケージとモデル依存関係をピンする
- ランダムシードを設定し、非決定的 GPU 挙動を文書化する
- データセットバージョン、コード SHA、設定ハッシュ、メトリクス、アーティファクト URI を記録する
- 前処理ロジックをノートブックとは別ではなくモデルアーティファクトと一緒に保存する
- トレーニング、評価、推論変換を共有または 1 つのソースから生成する
- 再試行がアーティファクトやメトリクスを破損しないようすべてのステップを冪等にする

不変値と純粋変換関数を優先する。フィーチャー生成中に共有データフレームやグローバル設定を変更することを避ける。

```python
import hashlib
from dataclasses import dataclass
from pathlib import Path


@dataclass(frozen=True)
class TrainingConfig:
    dataset_uri: str
    model_dir: Path
    seed: int
    learning_rate: float
    batch_size: int


def artifact_name(config: TrainingConfig, code_sha: str) -> str:
    config_key = f"{config.dataset_uri}:{config.seed}:{config.learning_rate}:{config.batch_size}"
    config_hash = hashlib.sha256(config_key.encode("utf-8")).hexdigest()[:12]
    return f"{code_sha[:12]}-{config_hash}"
```

### 4. 昇格前に評価する

昇格基準はトレーニング完了前に宣言されるべきである:

- ベースラインモデルと現在の本番モデルの比較
- 製品挙動に合わせた主要メトリック
- レイテンシ、キャリブレーション、公平性スライス、コスト、エラー集中のためのガードレールメトリクス
- 重要なコホート、地理、デバイス、言語、データソースのためのスライスメトリクス
- メトリクスがノイジーな場合の信頼区間または繰り返しラン分散
- 高インパクトモデルの場合はヒューマンレビュー済みの失敗例
- 明示的な「出荷しない」閾値

```python
PROMOTION_GATES = {
    "auc": ("min", 0.82),
    "calibration_error": ("max", 0.04),
    "p95_latency_ms": ("max", 80),
}


def assert_promotion_ready(metrics: dict[str, float]) -> None:
    missing = sorted(name for name in PROMOTION_GATES if name not in metrics)
    if missing:
        raise ValueError(f"Model promotion metrics missing required gates: {missing}")

    failures = {
        name: value
        for name, (direction, threshold) in PROMOTION_GATES.items()
        for value in [metrics[name]]
        if (direction == "min" and value < threshold)
        or (direction == "max" and value > threshold)
    }
    if failures:
        raise ValueError(f"Model failed promotion gates: {failures}")
```

オフラインメトリクスをゲートとして使う。保証ではない。モデルが製品挙動を変える場合、フルロールアウト前にシャドー評価、カナリアロールアウト、または A/B テストを計画する。

### 5. サービング用にパッケージ

ML アーティファクトは、サービング契約がテスト可能なときにのみ本番対応である:

- モデルアーティファクトはバージョン、トレーニングデータ参照、設定、前処理を含む
- 入力スキーマは無効、古い、または範囲外のフィーチャーを拒否する
- 出力スキーマは有用な場合モデルバージョンと信頼度または説明フィールドを含む
- サービングパスはタイムアウト、バッチング、リソース制限、フォールバック挙動を持つ
- CPU/GPU 要件は明示的でテストされる
- 予測ログは PII を避け、デバッグとラベル結合のための十分な識別子を含む
- 統合テストは欠落フィーチャー、古いフィーチャー、不正な型、空のバッチ、フォールバックパスをカバーする

トレーニング専用フィーチャーコードを、等価性を証明するテストなしにサービングフィーチャーコードから分岐させない。

### 6. モデルを運用する

モデルモニタリングはシステムと品質の両方のシグナルを必要とする:

- 可用性、エラー率、タイムアウト率、キュー深度、p50/p95/p99 レイテンシ
- フィーチャー null 率、範囲ドリフト、カテゴリドリフト、鮮度ドリフト
- 予測分布ドリフトと信頼度分布ドリフト
- ラベル到着健全性と遅延品質メトリクス
- ビジネス KPI ガードレールとロールバックトリガー
- カナリアとロールバックのためのバージョンごとのダッシュボード

すべてのデプロイには、前のアーティファクト、設定、データ依存関係、トラフィック切り替えメカニズムを命名するロールバック計画が必要である。

## レビューチェックリスト

- [ ] 予測契約が明示的でテスト可能
- [ ] データ契約がエンティティ粒度、ラベルタイミング、フィーチャータイミング、スナップショット/バージョンを定義
- [ ] リークリスクが予測時可用性に対してチェックされた
- [ ] トレーニングがコード、設定、データバージョン、シードから再現可能
- [ ] メトリクスがベースラインと現在の本番モデルと比較される
- [ ] 高リスクコホート用のスライスメトリクスとガードレールが含まれる
- [ ] 昇格ゲートが自動化されフェイルクローズ
- [ ] トレーニングとサービング変換が共有または等価性テスト済み
- [ ] モデルアーティファクトがバージョン、設定、データセット参照、前処理を持つ
- [ ] サービングパスが入力を検証し、タイムアウト、フォールバック、ロールバック挙動を持つ
- [ ] モニタリングがシステム健全性、フィーチャードリフト、予測ドリフト、遅延ラベルをカバー
- [ ] 機密データがアーティファクト、ログ、プロンプト、例から除外される

## アンチパターン

- ノートブック状態がモデル再現に必要
- ランダム分割が将来データを検証またはテストセットにリーク
- フィーチャー結合がイベント時間とラベル可用性を無視
- 重要なスライスが回帰する一方でオフラインメトリックが改善
- 閾値がテストセットで繰り返しチューニングされる
- トレーニング前処理がサービングコードに手動でコピーされる
- モデルバージョンが予測ログから欠落
- モニタリングがサービスアップタイムのみチェックし、データや予測品質をチェックしない
- ロールバックが既知良好アーティファクトへの切り替えではなく再トレーニングを必要とする

## 出力の期待

このスキルを使う際は具体的なアーティファクトを返す: データ契約、昇格ゲート、パイプラインステップ、テスト計画、デプロイ計画、レビュー所見。本番準備をブロックする未知数を仮定で埋めるのではなく、それらを明示する。
