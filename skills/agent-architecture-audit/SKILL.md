# エージェントアーキテクチャ監査

ラッパー層・古いメモリ・リトライループ・トランスポート/レンダリングの変更の背後に障害を隠すエージェントシステム向けの診断ワークフローである。

## 起動するタイミング

**以下の場合は必須:**
- エージェントまたは LLM 駆動アプリケーションを本番環境にリリースする時
- ツール呼び出し・メモリ・マルチステップワークフローを伴う機能をリリースする時
- ラッパー層追加後にエージェントの挙動が劣化した時
- ユーザーが「エージェントが悪化している」「ツールが不安定」と報告する時
- 同じモデルがプレイグラウンドでは動くがラッパー内で壊れる時
- 根本原因を見つけられないまま 15 分以上エージェント挙動をデバッグしている時

**特に重要なケース:**
- 新しいプロンプト層・ツール定義・メモリシステムを追加した時
- システム内の異なるエージェントが一貫しない挙動を示す時
- モデルが昨日まで正常で、今日ハルシネーションする時
- 応答を密かに変更している隠れた修復/リトライループが疑われる時

**以下には使用しない:**
- 一般的なコードデバッグ — `agent-introspection-debugging` を使う
- コードレビュー — 言語固有のレビューエージェントを使う
- セキュリティスキャン — `security-review` または `security-review/scan` を使う
- エージェントパフォーマンスベンチマーク — `agent-eval` を使う
- 新機能の開発 — 適切なワークフロースキルを使う

## 12 層スタック

すべてのエージェントシステムにはこれらの層が存在する。いずれもが回答を破壊しうる:

| # | 層 | 何が起こりうるか |
|---|-------|----------------|
| 1 | System prompt | 矛盾した指示、指示の肥大化 |
| 2 | Session history | 前回ターンからの古い文脈注入 |
| 3 | Long-term memory | セッションをまたいだ汚染、新しい会話に古いトピック |
| 4 | Distillation | 圧縮された成果物が擬似事実として再侵入 |
| 5 | Active recall | 冗長な再要約層が文脈を浪費 |
| 6 | Tool selection | 誤ったツールルーティング、モデルが必須ツールをスキップ |
| 7 | Tool execution | ハルシネーション実行 — 呼び出したと主張するが実行していない |
| 8 | Tool interpretation | ツール出力の誤読または無視 |
| 9 | Answer shaping | 最終応答のフォーマット破壊 |
| 10 | Platform rendering | トランスポート層変更 (UI・API・CLI が有効な回答を変えてしまう) |
| 11 | Hidden repair loops | 第二の LLM パスを走らせる隠れたフォールバック/リトライエージェント |
| 12 | Persistence | 期限切れの状態やキャッシュされた成果物が生きた証拠として再利用される |

## よくある失敗パターン

### 1. ラッパー回帰

ベースモデルは正しい回答を生成するが、ラッパー層が悪化させる。

**症状:**
- モデルはプレイグラウンドや直接 API 呼び出しでは動くが、エージェント内で壊れる
- 新しいプロンプト層を追加したら既存挙動が劣化した
- エージェントは自信ありげに聞こえるが自信を持って間違っている
- 「前回の更新前は動いていた」

### 2. メモリ汚染

履歴・メモリ取得・蒸留を介して古いトピックが新しい会話に漏れ出す。

**症状:**
- エージェントが無関係な過去のトピックを持ち出す
- ユーザーの修正が定着しない (古いメモリが新しいものを上書きする)
- 同セッションの成果物が擬似事実として再侵入する
- メモリが際限なく成長し、応答品質が時間とともに劣化する

### 3. ツール規律の失敗

ツールがプロンプトで宣言されているがコードで強制されていない。モデルがスキップしたり実行をハルシネーションする。

**症状:**
- プロンプトに「ツール X を必ず使用」とあるが、モデルが呼び出さずに回答する
- ツール結果が正しく見えるが実際には実行されていない
- 異なるツールが同じ責務を奪い合う
- モデルが使うべきでない時にツールを使い、使うべき時にスキップする

### 4. レンダリング/トランスポートの破壊

エージェント内部の回答は正しいが、プラットフォーム層が配信中に変更してしまう。

**症状:**
- ログでは正しい回答が示されるが、ユーザーには壊れた出力が見える
- マークダウンレンダリング・JSON パース・ストリーミング断片が有効な応答を破壊する
- 隠れたフォールバックエージェントが配信前に静かに回答を置き換える
- 出力がターミナルと UI で異なる

### 5. 隠れたエージェント層

明示的な契約なしで静かな修復・リトライ・要約・想起エージェントが走る。

**症状:**
- 内部生成とユーザー配信の間で出力が変化する
- 「Auto-fix」ループが、ユーザーが知らない第二の LLM パスを実行する
- 複数のエージェントが調整なしで同じ出力を変更する
- 目に見えない層によって回答が「滑らかにされる」または「修正される」

## 監査ワークフロー

### Phase 1: スコープ

何を監査するかを定義する:

- **対象システム** — どのエージェントアプリケーションか?
- **エントリポイント** — ユーザーはどのように利用するか?
- **モデルスタック** — どの LLM・プロバイダーか?
- **症状** — ユーザーは何を報告しているか?
- **時間枠** — いつから始まったか?
- **監査する層** — 12 層のうちどれが該当するか?

### Phase 2: 証拠収集

コードベースから証拠を収集する:

- **ソースコード** — エージェントループ・ツールルーター・メモリ admission・プロンプト組み立て
- **ログ** — 過去のセッショントレース・ツール呼び出し記録
- **設定** — プロンプトテンプレート・ツールスキーマ・プロバイダー設定
- **メモリファイル** — SOP・知識ベース・セッションアーカイブ

アンチパターンの検索には `rg` を使用する:

```bash
# Tool requirements expressed only in prompt text (not code)
rg "must.*tool|必须.*工具|required.*call" --type md

# Tool execution without validation
rg "tool_call|toolCall|tool_use" --type py --type ts

# Hidden LLM calls outside main agent loop
rg "completion|chat\.create|messages\.create|llm\.invoke"

# Memory admission without user-correction priority
rg "memory.*admit|long.*term.*update|persist.*memory" --type py --type ts

# Fallback loops that run additional LLM calls
rg "fallback|retry.*llm|repair.*prompt|re-?prompt" --type py --type ts

# Silent output mutation
rg "mutate|rewrite.*response|transform.*output|shap" --type py --type ts
```

### Phase 3: 失敗マッピング

所見ごとに以下を文書化する:

- **症状** — ユーザーが見るもの
- **メカニズム** — ラッパーがそれを引き起こす仕組み
- **発生層** — 12 層のどれか
- **根本原因** — 最も深い原因
- **証拠** — file:line または log:row 参照
- **信頼度** — 0.0 から 1.0

### Phase 4: 修正戦略

デフォルトの修正順序 (プロンプトファーストではなくコードファースト):

1. **ツール要件をコードで強制する** — プロンプトテキストだけでなくコードで強制する
2. **隠れた修復エージェントを削除または限定する** — フォールバックを契約付きで明示化する
3. **文脈の重複を削減する** — プロンプト + 履歴 + メモリ + 蒸留に同じ情報がある
4. **メモリ admission を厳格化する** — ユーザー修正 > エージェント主張
5. **蒸留トリガーを厳格化する** — 圧縮すべきでないものを圧縮しない
6. **レンダリングの変更を減らす** — 変換せずパススルーする
7. **型付き JSON エンベロープに変換する** — 自由形式の散文ではなく構造化された内部フロー

## 重大度モデル

| レベル | 意味 | 対応 |
|-------|---------|--------|
| `critical` | エージェントが自信を持って誤った運用挙動を生成しうる | 次のリリース前に修正 |
| `high` | エージェントが頻繁に正確性や安定性を低下させる | 今スプリントで修正 |
| `medium` | 正確性は通常維持されるが出力が脆弱または無駄 | 次のサイクルで計画 |
| `low` | 主に表面的または保守性の問題 | バックログ |

## 出力フォーマット

所見をユーザーに以下の順で提示する:

1. **重大度ランク付きの所見** (最も致命的なものから)
2. **アーキテクチャ診断** (どの層が何を、なぜ破壊したか)
3. **順序付き修正計画** (プロンプトファーストではなくコードファースト)

褒め言葉や要約で始めない。システムが壊れているなら、直接そう述べる。

## 簡易診断質問

エージェントシステムを監査する際、これらに答える:

| # | 質問 | Yes なら → |
|---|----------|----------|
| 1 | モデルは必須ツールをスキップしても回答できるか? | ツールがコードゲートされていない |
| 2 | 過去の会話内容が新しいターンに現れるか? | メモリ汚染 |
| 3 | 同じ情報がシステムプロンプトとメモリと履歴に存在するか? | 文脈の重複 |
| 4 | プラットフォームが配信前に第二の LLM パスを実行するか? | 隠れた修復ループ |
| 5 | 出力が内部生成とユーザー配信の間で異なるか? | レンダリング破壊 |
| 6 | 「ツール X を必ず使う」ルールがプロンプトテキストのみにあるか? | ツール規律の失敗 |
| 7 | エージェント自身の独白が永続的メモリになりうるか? | メモリ汚染 |

## 避けるべきアンチパターン

- ラッパー層の回帰を反証する前にモデルのせいにすることを避ける。
- 汚染経路を示すことなくメモリのせいにすることを避ける。
- きれいな現在の状態が、汚れた過去の事件を消し去ることを許さない。
- マークダウンの散文を信頼できる内部プロトコルとして扱わない。
- コードが強制しないのに「ツールを必ず使え」というプロンプトテキストを受け入れない。
- 所見は直接的で、証拠に基づき、重大度ランク付きに保つ。
