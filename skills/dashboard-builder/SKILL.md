# Dashboard Builder

タスクが人々が運用できるダッシュボードを構築することのときに使う。

目標は「すべてのメトリクスを表示」ではない。目標は以下に答えること:

- 健全か?
- ボトルネックはどこか?
- 何が変わったか?
- 誰かがどのアクションを取るべきか?

## 利用するタイミング

- 「Kafka モニタリングダッシュボードを構築」
- 「Elasticsearch 用 Grafana ダッシュボードを作成」
- 「このサービス用 SigNoz ダッシュボードを作る」
- 「このメトリクスリストを実運用ダッシュボードに変換」

## ガードレール

- ビジュアルレイアウトから始めない。オペレータ質問から始める
- 存在するという理由だけですべての利用可能メトリクスを含めない
- 健全・スループット・リソースパネルを構造なしで混ぜない
- タイトル・単位・妥当なしきい値なしのパネルをリリースしない

## ワークフロー

### 1. 運用質問を定義する

以下を中心に整理する:

- 健全性 / 可用性
- レイテンシ / パフォーマンス
- スループット / ボリューム
- 飽和 / リソース
- サービス固有リスク

### 2. ターゲットプラットフォームスキーマを学習

最初に既存ダッシュボードを検査する:

- JSON 構造
- クエリ言語
- 変数
- しきい値スタイリング
- セクションレイアウト

### 3. 最小限有用なボードを構築

推奨構造:

1. 概要
2. パフォーマンス
3. リソース
4. サービス固有セクション

### 4. バニティパネルをカット

各パネルは実質問に答えるべき。そうでなければ削除する。

## 例パネルセット

### Elasticsearch

- クラスタ健全性
- シャード割り当て
- 検索レイテンシ
- インデックスレート
- JVM ヒープ / GC

### Kafka

- ブローカー数
- アンダーレプリケートパーティション
- メッセージイン / アウト
- コンシューマラグ
- ディスクとネットワーク圧

### API gateway / ingress

- リクエストレート
- p50 / p95 / p99 レイテンシ
- エラーレート
- アップストリーム健全性
- アクティブ接続

## 品質チェックリスト

- [ ] 有効なダッシュボード JSON
- [ ] 明確なセクショングループ化
- [ ] タイトルと単位が存在
- [ ] しきい値/ステータスカラーが意味ある
- [ ] よくあるフィルター用に変数が存在
- [ ] デフォルト時間範囲とリフレッシュが妥当
- [ ] オペレータ価値のないバニティパネルがない
