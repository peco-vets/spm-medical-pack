# Customer Billing Ops

このスキルを汎用支払い API 設計ではなく実際の顧客運用に使う。

目標はオペレータが答えるのを助けることである: この顧客は誰か、何が起きたか、最も安全な修正は何か、どのフォローアップを送るべきか?

## 利用するタイミング

- 顧客が請求が壊れている、返金が欲しい、キャンセルできないと言う
- 重複サブスクリプション、誤課金、失敗した更新、解約リスクを調査
- プランミックス、アクティブサブスクリプション、年次 vs 月次転換、チームシート混乱をレビュー
- 請求ポータルフローの作成または検証
- サブスクリプション、請求書、返金、支払い方法に触れるサポート不満の監査

## 優先ツールサーフェス

- Stripe のような接続された請求ツールを最初に使う
- メール、GitHub、issue トラッカーは補助証拠としてのみ使う
- プラットフォームが必要なコントロールを既に提供しているとき、カスタムアカウント管理コードよりホスト型請求/顧客ポータルを優先

## ガードレール

- 応答にシークレットキー、完全なカード詳細、不要な顧客 PII を露出しない
- やみくもに返金しない。最初に問題を分類する
- 以下を区別する:
  - 偶発的な重複購入
  - 意図的なマルチシートまたはチーム購入
  - 壊れた製品 / 未達価値
  - 失敗したまたは不完全なチェックアウト
  - 欠落セルフサーブコントロールによるキャンセル
- 年次プラン、チームプラン、按分状態では、アクション前に契約形状を検証する

## ワークフロー

### 1. 顧客をクリーンに特定する

利用可能な最強の識別子から始める:

- 顧客メール
- Stripe 顧客 ID
- サブスクリプション ID
- 請求書 ID
- 請求にマップバックすると分かっている場合の GitHub ユーザー名やサポートメール

簡潔なアイデンティティサマリを返す:

- 顧客
- アクティブサブスクリプション
- キャンセルされたサブスクリプション
- 請求書
- 重複アクティブサブスクリプションなどの明らかな異常

### 2. 問題を分類する

行動する前にケースを 1 つのバケットに入れる:

| ケース | 典型的アクション |
|------|----------------|
| 重複個人サブスクリプション | 余分をキャンセル、返金を検討 |
| 実際のマルチシート/チーム意図 | シートを保持、請求モデルを明確化 |
| 失敗した支払い / 不完全なチェックアウト | ポータルまたは支払い方法更新で回復 |
| 欠落セルフサーブコントロール | ポータル、キャンセルパス、請求書アクセスを提供 |
| 製品失敗または信頼破綻 | 返金、謝罪、製品問題をログ |

### 3. 最も安全な可逆アクションを最初に取る

優先順序:

1. セルフサーブ管理を復元
2. 重複または壊れた請求状態を修正
3. 影響を受けた課金または重複のみ返金
4. 理由を文書化
5. 短い顧客フォローアップを送信

修正が製品作業を必要とするなら、分離する:

- 顧客修復は今
- 製品バグ / ワークフローギャップはバックログ

### 4. オペレータ側の製品ギャップをチェック

顧客の痛みが欠落したオペレータサーフェスから来ているなら、明示的に呼び出す。よくある例:

- 請求ポータルなし
- 使用量/レート制限可視性なし
- プラン/シート説明なし
- キャンセルフローなし
- 重複サブスクリプションガードなし

これらをサポートインシデントだけでなく ECC または Web サイトのフォローアップ項目として扱う。

### 5. オペレータハンドオフを生成

以下で終わる:

- 顧客状態サマリ
- 取られたアクション
- 収益影響
- 送るフォローアップテキスト
- 作成する製品またはバックログ issue

## 出力フォーマット

この構造を使う:

```text
CUSTOMER
- name / email
- relevant account identifiers

BILLING STATE
- active subscriptions
- invoice or renewal state
- anomalies

DECISION
- issue classification
- why this action is correct

ACTION TAKEN
- refund / cancel / portal / no-op

FOLLOW-UP
- short customer message

PRODUCT GAP
- what should be fixed in the product or website
```

## 良い推奨の例

- 「正しい修正は請求ポータルで、まだカスタムダッシュボードではない」
- 「これは実際のチームシート購入ではなく、重複個人チェックアウトに見える」
- 「重複課金 1 つを返金、残りのアクティブサブスクリプションを維持、必要なら後で顧客を組織課金に転換」
