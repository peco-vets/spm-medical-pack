# Security Bounty Hunter

責任ある開示またはバウンティ提出のための実用的な脆弱性発見が目的であり、広範なベストプラクティスレビューではないときに使う。

## 使用するタイミング

- リポジトリの悪用可能な脆弱性のスキャン
- Huntr、HackerOne、または類似のバウンティ提出の準備
- 「これは本当に支払われるか？」が問いであるトリアージ（「これは理論的に安全でないか？」ではない）

## 動作の仕組み

リモート到達可能でユーザー制御可能な攻撃パスを優先し、プラットフォームが情報レベルまたはスコープ外として日常的に拒否するパターンを捨てる。

## スコープ内パターン

これらは一貫して重要な問題のタイプ：

| パターン | CWE | 典型的影響 |
| --- | --- | --- |
| ユーザー制御 URL 経由の SSRF | CWE-918 | 内部ネットワークアクセス、クラウドメタデータ窃取 |
| ミドルウェアまたは API ガードの認証バイパス | CWE-287 | 不正なアカウントまたはデータアクセス |
| リモートデシリアライゼーションまたはアップロードから RCE パス | CWE-502 | コード実行 |
| 到達可能なエンドポイントの SQL インジェクション | CWE-89 | データ漏洩、認証バイパス、データ破壊 |
| リクエストハンドラのコマンドインジェクション | CWE-78 | コード実行 |
| ファイル提供パスのパストラバーサル | CWE-22 | 任意ファイル読取または書込 |
| 自動トリガー XSS | CWE-79 | セッション窃取、管理者侵害 |

## スキップするもの

これらは通常、プログラムが別途明記しない限り低シグナルまたはバウンティスコープ外：

- リモートパスのないローカルのみの `pickle.loads`、`torch.load`、または同等
- CLI のみのツールでの `eval()` または `exec()`
- 完全にハードコードされたコマンドでの `shell=True`
- セキュリティヘッダー欠落そのもの
- 悪用影響のない一般的なレート制限苦情
- 被害者にコードを手動で貼り付けさせる必要のあるセルフ XSS
- ターゲットプログラムスコープの一部ではない CI/CD インジェクション
- デモ、例、またはテストのみのコード

## ワークフロー

1. スコープを先にチェック：プログラムルール、SECURITY.md、開示チャネル、除外
2. 実際のエントリポイントを見つける：HTTP ハンドラ、アップロード、バックグラウンドジョブ、Webhook、パーサ、統合エンドポイント
3. 役立つ場所で静的ツールを実行するが、トリアージ入力としてのみ扱う
4. 実コードパスをエンドツーエンドで読む
5. ユーザー制御が意味のあるシンクに到達することを証明する
6. 可能な限り小さな安全な PoC で悪用可能性と影響を確認する
7. レポート作成前に重複をチェックする

## トリアージループの例

```bash
semgrep --config=auto --severity=ERROR --severity=WARNING --json
```

次に手動フィルタ：

- テスト、デモ、フィクスチャ、ベンダーコードを捨てる
- ローカルのみまたは非到達パスを捨てる
- 明確なネットワークまたはユーザー制御ルートを持つ発見のみ保持

## レポート構造

```markdown
## Description
[What the vulnerability is and why it matters]

## Vulnerable Code
[File path, line range, and a small snippet]

## Proof of Concept
[Minimal working request or script]

## Impact
[What the attacker can achieve]

## Affected Version
[Version, commit, or deployment target tested]
```

## 品質ゲート

提出前に：

- コードパスが実ユーザーまたはネットワーク境界から到達可能
- 入力が真にユーザー制御である
- シンクが意味があり悪用可能である
- PoC が動作する
- 問題がアドバイザリ、CVE、またはオープンチケットですでにカバーされていない
- ターゲットが実際にバウンティプログラムのスコープ内である
