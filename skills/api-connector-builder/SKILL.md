# API コネクタビルダー

汎用 HTTP クライアントではなく、リポジトリ固有の統合サーフェスを追加するのが仕事のときに使う。

要点はホストリポジトリのパターンに合わせることである:

- コネクタレイアウト
- 設定スキーマ
- 認証モデル
- エラー処理
- テストスタイル
- 登録/発見の配線

## 利用するタイミング

- 「このプロジェクトに Jira コネクタを作って」
- 「既存パターンに従って Slack プロバイダを追加して」
- 「この API のための新しい統合を作って」
- 「リポジトリのコネクタスタイルに合うプラグインを構築して」

## ガードレール

- 既に統合アーキテクチャがあるリポジトリで新しいものを発明しない
- ベンダードキュメントだけから始めない。まず既存のリポジトリ内コネクタから始める
- リポジトリがレジストリ配線・テスト・ドキュメントを期待しているなら、トランスポートコードで止まらない
- リポジトリに新しい現行パターンがある場合、古いコネクタをカーゴカルト的に真似ない

## ワークフロー

### 1. ハウススタイルを学ぶ

少なくとも 2 つの既存コネクタ/プロバイダを検査し以下をマップする:

- ファイルレイアウト
- 抽象化境界
- 設定モデル
- リトライ / ページネーション規約
- レジストリフック
- テストフィクスチャと命名

### 2. ターゲット統合を絞り込む

リポジトリが実際に必要とするサーフェスのみ定義する:

- 認証フロー
- 主要エンティティ
- 中核読み/書き操作
- ページネーションとレート制限
- Webhook またはポーリングモデル

### 3. リポジトリネイティブな層で構築する

典型的なスライス:

- 設定/スキーマ
- クライアント/トランスポート
- マッピング層
- コネクタ/プロバイダエントリポイント
- 登録
- テスト

### 4. ソースパターンに対して検証する

新しいコネクタはコードベース内で自明に見えるべきで、別のエコシステムから持ち込んだように見えてはならない。

## 参照シェイプ

### プロバイダスタイル

```text
providers/
  existing_provider/
    __init__.py
    provider.py
    config.py
```

### コネクタスタイル

```text
integrations/
  existing/
    client.py
    models.py
    connector.py
```

### TypeScript プラグインスタイル

```text
src/integrations/
  existing/
    index.ts
    client.ts
    types.ts
    test.ts
```

## 品質チェックリスト

- [ ] 既存のリポジトリ内統合パターンに合致する
- [ ] 設定バリデーションが存在する
- [ ] 認証とエラー処理が明示的である
- [ ] ページネーション/リトライ挙動がリポジトリ規範に従う
- [ ] レジストリ/発見の配線が完了している
- [ ] テストがホストリポジトリのスタイルを反映する
- [ ] リポジトリが期待するならドキュメント/例が更新されている
