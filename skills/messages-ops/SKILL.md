# Messages Ops

ライブメッセージ取得のタスクにはこれを使う: iMessage、DM、最近のワンタイムコード、フォローアップ前のスレッド検査。

これはメール作業ではない。主要なサーフェスがメールボックスの場合は `email-ops` を使う。

## スキルスタック

関連する場合、これらの ECC ネイティブスキルをワークフローに引き入れる:

- メッセージタスクが実はメールボックス作業の場合は `email-ops`
- DM スレッドがアウトバウンドネットワーク作業に属する場合は `connections-optimizer`
- ライブスレッドがターゲティングまたはウォームパスアウトリーチに情報を提供すべき場合は `lead-intelligence`
- スレッド内容を永続コンテキストにキャプチャする必要がある場合は `knowledge-ops`

## 使用するタイミング

- ユーザーが「メッセージを読む」「テキストを確認」「DM を見る」または「コードを見つける」と発言した場合
- タスクがライブスレッドまたはローカルメッセージングサーフェスに配信された最近のコードに依存する場合
- ユーザーがどのソースやスレッドが検査されたかの証明を求めている場合

## ガードレール

- まずソースを解決する:
  - ローカルメッセージ
  - X / ソーシャル DM
  - 別のブラウザゲート付きメッセージサーフェス
- ソースを名指しせずにスレッドが確認されたと主張しない
- 検証済みヘルパーや標準パスが存在する場合、生のデータベースアクセスを即興で行わない
- 認証や MFA がサーフェスをブロックしている場合、正確なブロッカーを報告する

## ワークフロー

### 1. 正確なスレッドを解決する

他の何かをする前に以下を確定する:

- メッセージサーフェス
- 送信者 / 受信者 / サービス
- 時間ウィンドウ
- タスクが取得、検査、または返信準備か

### 2. ドラフト前に読む

タスクがアウトバウンドのフォローアップに発展する可能性がある場合:

- 最新のインバウンドを読む
- オープンループを特定する
- 必要に応じて正しいアウトバウンドスキルに引き継ぐ

### 3. コードはフォーカスされた取得タスクとして扱う

ワンタイムコードの場合:

- まず最近のローカルメッセージウィンドウを検索
- 可能な場合はサービスまたは送信者で絞り込む
- コードが見つかるかフォーカス検索が尽きるまで停止する

### 4. 正確な証拠を報告する

返却内容:

- 使用したソース
- 可能な場合はスレッドまたは送信者
- 時間ウィンドウ
- 正確なステータス:
  - read
  - code-found
  - blocked
  - awaiting reply draft

## 出力形式

```text
SOURCE
- message surface
- sender / thread / service

RESULT
- message summary or code
- time window

STATUS
- read / code-found / blocked / awaiting reply draft
```

## 落とし穴

- メールボックス作業と DM/テキスト作業を曖昧にしない
- ソースを名指しせずに取得を主張しない
- 依頼が最近のコード検索なのに広範な検索に時間を費やさない
- ブロッカーを露わにせずブロックされた認証パスを再試行し続けない

## 検証

- レスポンスがメッセージソースを名指ししている
- レスポンスに送信者、サービス、スレッド、または明確なブロッカーが含まれている
- 最終状態が明示的で境界づけられている
