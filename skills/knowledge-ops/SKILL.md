# Knowledge Operations

複数のストアにわたるナレッジを取り込み、整理、同期、取得するための多層ナレッジシステムを管理する。

ライブワークスペースモデルを優先する。
- コード作業は実際のクローン済みリポジトリ内に存在する
- アクティブな実行コンテキストは GitHub、Linear、およびリポジトリ内のワーキングコンテキストファイル内に存在する
- より広範な人間向けノートは、リポジトリ外のコンテキスト/アーカイブフォルダに置ける
- マシン横断の永続メモリは、シャドウリポジトリワークスペースではなくナレッジベースに属する

## 起動するタイミング

- ユーザーがナレッジベースに情報を保存したい場合
- 構造化ストレージにドキュメント、会話、データを取り込む場合
- システム間でナレッジを同期する場合 (ローカルファイル、MCP メモリ、Supabase、Git リポジトリ)
- 既存ナレッジの重複排除や整理を行う場合
- ユーザーが「KB に保存」「ナレッジ同期」「X について何を知っているか」「これを取り込む」「ナレッジベースを更新」と発言した場合
- 単純なメモリ想起を超えるあらゆるナレッジ管理タスク

## ナレッジアーキテクチャ

### レイヤー 1: アクティブな実行の真実
- **ソース:** GitHub の issue、PR、ディスカッション、リリースノート、Linear の issue/プロジェクト/ドキュメント
- **用途:** 作業の現在の運用状態
- **ルール:** アクティブなエンジニアリング計画、ロードマップ、ロールアウト、リリースに影響を与える場合は、まずここに置くことを優先する

### レイヤー 2: Claude Code メモリ (クイックアクセス)
- **パス:** `~/.claude/projects/*/memory/`
- **形式:** フロントマター付き Markdown ファイル
- **種類:** ユーザー設定、フィードバック、プロジェクトコンテキスト、リファレンス
- **用途:** 会話間で永続するクイックアクセス用コンテキスト
- **セッション開始時に自動ロードされる**

### レイヤー 3: MCP メモリサーバー (構造化ナレッジグラフ)
- **アクセス:** MCP メモリツール (create_entities、create_relations、add_observations、search_nodes)
- **用途:** 保存されたすべてのメモリ横断のセマンティック検索、リレーションマッピング
- **クエリ可能なグラフ構造によるセッション横断永続化**

### レイヤー 4: ナレッジベースリポジトリ / 永続ドキュメントストア
- **用途:** キュレートされた永続ノート、セッションエクスポート、合成リサーチ、オペレータメモリ、長文ドキュメント
- **ルール:** これはリポジトリ所有のコード以外のコンテンツに対するマシン横断コンテキストの優先永続ストアである

### レイヤー 5: 外部データストア (Supabase、PostgreSQL など)
- **用途:** 構造化データ、大規模ドキュメントストレージ、全文検索
- **適している場合:** メモリファイルには大きすぎるドキュメント、SQL クエリを必要とするデータ

### レイヤー 6: ローカルコンテキスト/アーカイブフォルダ
- **用途:** 人間向けノート、アーカイブされたゲームプラン、ローカルメディア整理、一時的な非コードドキュメント
- **ルール:** 情報保存には書き込み可能だが、シャドウコードワークスペースではない
- **避けるべき用途:** 上流に存在すべきアクティブなコード変更やリポジトリの真実

## 取り込みワークフロー

新しいナレッジを取り込む必要がある場合:

### 1. 分類
どのタイプのナレッジか?
- ビジネス決定 -> メモリファイル (プロジェクトタイプ) + MCP メモリ
- アクティブなロードマップ / リリース / 実装状態 -> まず GitHub + Linear
- 個人設定 -> メモリファイル (user/feedback タイプ)
- 参考情報 -> メモリファイル (reference タイプ) + MCP メモリ
- 大規模ドキュメント -> 外部データストア + メモリにサマリー
- 会話/セッション -> ナレッジベースリポジトリ + メモリに短いサマリー

### 2. 重複排除
このナレッジが既に存在するか確認する。
- 既存エントリをメモリファイルで検索
- 関連語で MCP メモリをクエリ
- 別のローカルノートを作成する前に、情報が既に GitHub や Linear に存在するか確認
- 重複を作らない。代わりに既存エントリを更新する

### 3. 保存
適切なレイヤーに書き込む。
- クイックアクセスのために Claude Code メモリを必ず更新
- セマンティック検索性とリレーションマッピングには MCP メモリを使用
- 情報がライブプロジェクトの真実を変更する場合は、まず GitHub / Linear を更新
- 永続的な長文追加にはナレッジベースリポジトリにコミット

### 4. インデックス
関連するインデックスやサマリーファイルを更新する。

## 同期操作

### 会話同期
会話履歴を定期的にナレッジベースに同期する。
- ソース: Claude セッションファイル、Codex セッション、その他のエージェントセッション
- 宛先: ナレッジベースリポジトリ
- ブラウズしやすいセッションインデックスを生成
- コミットしてプッシュ

### ワークスペース状態同期
重要なワークスペース設定とスクリプトをナレッジベースにミラーリング。
- ディレクトリマップを生成
- コミット前に機密設定をリダクション
- 時間経過の変更を追跡
- ナレッジベースやアーカイブフォルダをライブコードワークスペースとして扱わない

### GitHub / Linear 同期
情報がアクティブな実行に影響する場合:
- 関連する GitHub issue、PR、ディスカッション、リリースノート、ロードマップスレッドを更新
- 作業に永続的な計画コンテキストが必要な場合は Linear にサポートドキュメントを添付
- 後でローカルノートをミラーリングするのは、まだ価値を加える場合のみ

### クロスソースナレッジ同期
複数ソースからナレッジを 1 か所に集約する。
- Claude/ChatGPT/Grok 会話エクスポート
- ブラウザブックマーク
- GitHub アクティビティイベント
- ステータスサマリー書き込み、コミットしてプッシュ

## メモリパターン

```
# Short-term: current session context
Use TodoWrite for in-session task tracking

# Medium-term: project memory files
Write to ~/.claude/projects/*/memory/ for cross-session recall

# Long-term: GitHub / Linear / KB
Put active execution truth in GitHub + Linear
Put durable synthesized context in the knowledge base repo

# Semantic layer: MCP knowledge graph
Use mcp__memory__create_entities for permanent structured data
Use mcp__memory__create_relations for relationship mapping
Use mcp__memory__add_observations for new facts about known entities
Use mcp__memory__search_nodes to find existing knowledge
```

## ベストプラクティス

- メモリファイルは簡潔に保つ。ファイルが無制限に肥大化するより、古いデータをアーカイブする
- すべてのナレッジファイルのメタデータにフロントマター (YAML) を使用する
- 保存前に重複排除する。まず検索してから作成または更新する
- ファクトセットごとに 1 つの正規ホームを優先する。同じ計画のローカルノート、リポジトリファイル、トラッカードキュメントへの並行コピーを避ける
- Git にコミットする前に機密情報 (API キー、パスワード) をリダクションする
- ナレッジファイルには一貫した命名規約 (lowercase-kebab-case) を使用する
- 取得しやすくするためにエントリをトピック/カテゴリでタグ付けする

## 品質ゲート

ナレッジ操作を完了する前に:
- 重複エントリを作成していない
- Git 追跡ファイルから機密データがリダクションされている
- インデックスとサマリーが更新されている
- データ型に適したストレージレイヤーが選択されている
- 関連するクロスリファレンスが追加されている
