# repo-scan

> エコシステムにはそれぞれ独自の依存関係マネージャーがあるが、C++、Android、iOS、Web を横断して「実際に自社コードはどれくらいか、第三者は何か、デッドウェイトは何か」を教えてくれるツールはない。

## 使用するタイミング

- 大規模なレガシーコードベースを引き継ぎ、構造概要が必要な場合
- 大規模なリファクタリングの前 — 何がコアで、何が重複で、何が dead か特定する
- ソースに直接埋め込まれた第三者依存関係を監査する（パッケージマネージャーで宣言されていないもの）
- モノレポ再編成のためのアーキテクチャ意思決定記録を準備する

## インストール

```bash
# Fetch only the pinned commit for reproducibility
mkdir -p ~/.claude/skills/repo-scan
git init repo-scan
cd repo-scan
git remote add origin https://github.com/haibindev/repo-scan.git
git fetch --depth 1 origin 2742664
git checkout --detach FETCH_HEAD
cp -r . ~/.claude/skills/repo-scan
```

> エージェントスキルをインストールする前にソースをレビューする。

## コア機能

| 機能 | 説明 |
|---|---|
| **クロススタックスキャン** | C/C++、Java/Android、iOS（OC/Swift）、Web（TS/JS/Vue）を 1 パスで |
| **ファイル分類** | 各ファイルをプロジェクトコード、第三者、ビルド成果物としてタグ付け |
| **ライブラリ検出** | バージョン抽出付きで 50+ の既知ライブラリ（FFmpeg、Boost、OpenSSL…） |
| **4 段階判定** | Core Asset / Extract & Merge / Rebuild / Deprecate |
| **HTML レポート** | ドリルダウンナビゲーション付きインタラクティブダークテーマページ |
| **モノレポサポート** | サマリ + サブプロジェクトレポート付き階層スキャン |

## 解析深度レベル

| レベル | 読まれるファイル | ユースケース |
|---|---|---|
| `fast` | モジュールあたり 1-2 | 巨大ディレクトリの素早いインベントリ |
| `standard` | モジュールあたり 2-5 | 完全な依存関係 + アーキテクチャチェック付きデフォルト監査 |
| `deep` | モジュールあたり 5-10 | スレッド安全性、メモリ管理、API 一貫性を追加 |
| `full` | 全ファイル | マージ前包括レビュー |

## 動作の仕組み

1. **リポサーフェスを分類**：ファイルを列挙し、各々をプロジェクトコード、埋め込み第三者コード、ビルド成果物としてタグ付けする。
2. **埋め込みライブラリを検出**：ディレクトリ名、ヘッダ、ライセンスファイル、バージョンマーカーを調べてバンドルされた依存関係と可能性のあるバージョンを特定する。
3. **各モジュールをスコアリング**：ファイルをモジュールまたはサブシステムでグループ化し、所有権、重複、メンテナンスコストに基づいて 4 つの判定の 1 つを割り当てる。
4. **構造リスクをハイライト**：デッドウェイト成果物、重複ラッパー、古いベンダーコード、抽出・再構築・廃止すべきモジュールを指摘する。
5. **レポートを生成**：簡潔なサマリと、モジュールごとのドリルダウン付きインタラクティブ HTML 出力を返し、監査を非同期にレビューできるようにする。

## 例

50,000 ファイルの C++ モノレポで：
- FFmpeg 2.x（2015 年のもの）がまだ本番で使われているのを発見
- 同じ SDK ラッパーが 3 回重複しているのを発見
- 636 MB のコミットされた Debug/ipch/obj ビルド成果物を特定
- 分類：プロジェクトコード 3 MB vs 第三者 596 MB

## ベストプラクティス

- 初回監査では `standard` 深度から始める
- 100+ モジュールのモノレポで素早いインベントリを取るには `fast` を使う
- リファクタリング用にフラグが立ったモジュールに `deep` を段階的に実行する
- サブプロジェクト横断の重複検出のためクロスモジュール解析をレビューする
