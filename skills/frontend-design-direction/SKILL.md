# フロントエンドデザイン方針

UI を機能させるだけでなく、目的的で洗練され、プロダクトドメインに適合する感触にする作業のときに用いる。

出典: コミュニティ PR #1659（`linus707` 著、stale）からのサルベージ。

備考: ECC は意図的に正規 Anthropic `frontend-design` スキルを再同梱していない。公式アップストリームスキルが欲しい場合は `anthropics/skills` からインストールすること。本スキルは #1659 の有用なローカルガイダンスを ECC 固有のデザイン方針としてサルベージしたものである。

## 利用タイミング

- ユーザーが Web ページ・アプリ・ダッシュボード・アーティファクト・コンポーネント・UI の構築を依頼
- ユーザーがインターフェースをより洗練・特徴的・美しく、汎用らしさを減らすことを依頼
- 実装が視覚階層・タイポグラフィ・色・モーション・レイアウト・操作の判断を必要とする
- 現状の UI は動作するが、フラット・汎用・テンプレ感・対象ミスマッチに見える

## デザイン方針

コーディング前に具体的方針を選ぶ。

1. 目的: インターフェースはどんな仕事をするか?
2. 対象: 誰がこのワークフローを反復し、最初に何を読み取るべきか?
3. トーン: utilitarian、editorial、playful、industrial、refined、technical、maximal、minimal、dense、calm、その他明示的方針
4. 記憶に残る詳細: 結果を意図的に感じさせる1つのデザインアイデア
5. 制約: フレームワーク・アクセシビリティ・パフォーマンス・レスポンシブ・既存デザインシステム

方針はドメインに合わせる。SaaS オペレーションツールは通常 dense・quiet・スキャナブル。ポートフォリオ・ローンチページ・ゲーム・エディトリアル作品はより表現的にできる。日常反復利用が必要なツールにランディングページ的構成を押し付けないこと。

## 実装ガイダンス

- ユーザーがマーケティングコピーを明示要求しない限り、実際に使える体験を最初の画面として構築する
- 新しいビジュアルシステムを導入する前に、既存プロジェクトのコンポーネント・トークン・アイコンライブラリ・ルーティングパターンを使う
- インターフェースが画像・商品・場所・人物・ゲームプレイ・チャート・点検可能メディアに依存する場合は、実アセットまたは生成ビジュアルアセットを使う
- 汎用の過大なヒーローテキストよりも、文脈的タイポグラフィとスペーシングを優先する
- パレットは多次元に保つ — 1色族が支配する UI を避ける
- CSS 変数または既存デザイントークンを使い、状態をまたいで方針が一貫するようにする
- レスポンシブ制約を明示設計する: grid・アスペクト比・min/max サイズ・安定ツールバー・固定フォーマットコントロールは、ラベルや hover 状態の出現で崩れてはならない
- モーションは控えめだが意図的に使う。装飾アニメーションよりも、状態を明確化する高シグナル遷移を優先する
- モバイルとデスクトップでテキスト適合を検証する。長いラベルはあふれず適切にラップ・リサイズする

## アンチパターン

- 一般的な生成パターン（紫グラデーション・装飾 blob・過大なカード・曖昧なヒーローコピー・ストック調の雰囲気メディア）をデフォルトにしない
- カード内にカード UI を入れない
- ドメインが抑制を要するときに装飾スタイルを至るところに使わない
- 主要プロダクト・ツール・対象・ワークフローを汎用マーケティングセクションの背後に隠さない
- コントロール自体で十分なときに装飾のための新規依存を追加しない
- UI の機能を UI 内で説明しない（コントロール自身が物語る）

## レビューチェックリスト

- 最初のビューポートがプロダクト・ワークフロー・対象を即座に伝える
- 視覚階層がスキャンと反復利用を支える
- タイポグラフィがコンテナに適合し、隣接コンテンツと重なっていない
- 色選択にコントラストがあり、単調なパレットに陥っていない
- 利用可能な場合、馴染みあるツール操作にアイコンを使う
- ボード・グリッド・ツールバー・コントロール・タイル・カウンターのレスポンシブレイアウトが安定寸法
- アセットがレンダリングされ、フィラーではなく主題を担っている
- モーションが方向感覚を改善し、遅さを隠していない
- 結果がリポジトリの既存フロントエンド慣習に合致する（明確な理由なく逸脱しない）
