# Unified Notifications Ops

実問題が pinging の欠落ではないときにこのスキルを使う。実問題は断片化された通知システム。

仕事は散在するイベントを 1 つのオペレータサーフェスに変えることで、以下を備える：
- 明確な重要度
- 明確な所有権
- 明確なルーティング
- 明確なフォローアップアクション

## 使用するタイミング

- ユーザーが GitHub、Linear、ローカルフック、デスクトップアラート、チャット、メール全体で統一された通知レーンを望む
- CI 失敗、レビュー依頼、issue 更新、オペレータイベントが切断された場所に到着
- 現在のセットアップがアクションではなくノイズを作る
- ユーザーが重複する通知ブランチまたはバックログ提案を 1 つの ECC ネイティブレーンに統合したい
- ワークスペースにすでにフック、MCP、または接続ツールがあるが、首尾一貫した通知ポリシーがない

## 推奨サーフェス

すでに存在するものから始める：
- GitHub issue、PR、レビュー、コメント、CI
- Linear issue/プロジェクト移動
- ローカルフックイベントとセッションライフサイクルシグナル
- デスクトップ通知プリミティブ
- 実際に存在する接続されたメール／チャットサーフェス

ユーザーに別個の通知製品を採用するよう伝えるよりも ECC ネイティブのオーケストレーションを推奨。

## 譲れないルール

- トークン、シークレット、Webhook シークレット、または内部識別子を決して公開しない
- 以下を分離：
  - イベントソース
  - 重要度
  - ルーティングチャネル
  - オペレータアクション
- 中断コストが不明なときはダイジェスト優先をデフォルトに
- すべてのイベントをすべてのチャネルにファンアウトしない
- 実修正がより良い issue トリアージ、フックポリシー、またはプロジェクトフローなら、明示的に言う

## イベントパイプライン

レーンを以下として扱う：

1. イベントを **Capture**
2. 緊急度と所有者を **Classify**
3. 正しいチャネルに **Route**
4. 重複と低シグナルチャーンを **Collapse**
5. 次のオペレータアクションを **Attach**

ゴールはより少ないより良い通知。

## デフォルト重要度モデル

| クラス | 例 | デフォルト処理 |
| --- | --- | --- |
| Critical | 壊れたデフォルトブランチ CI、セキュリティ問題、ブロックされたリリース、失敗したデプロイ | 今すぐ中断 |
| High | レビュー依頼、失敗 PR、オーナーブロッキングハンドオフ | 同日アラート |
| Medium | issue 状態変更、注目すべきコメント、バックログ移動 | ダイジェストまたはキュー |
| Low | 繰り返し成功、ルーティーンチャーン、冗長なライフサイクルマーカー | 抑制または折り畳み |

ワークスペースに重要度モデルがなければ、自動化を提案する前に構築する。

## ワークフロー

### 1. 現在のサーフェスをインベントリ

リスト：
- イベントソース
- 現在のチャネル
- アラートを発する既存のフック／スクリプト
- 同じイベントの重複パス
- 重要なものが表面化していないサイレント失敗ケース

ECC がすでに所有しているものを指摘。

### 2. 何が中断に値するかを決定

各イベントファミリーについて、答える：
- 誰が知る必要があるか？
- どれだけ速く知る必要があるか？
- これは中断、バッチ、または単にログすべきか？

これらのデフォルトを使う：
- リリース、CI、セキュリティ、オーナーブロッキングイベントには中断
- 中シグナル更新にはダイジェスト
- テレメトリと低シグナルライフサイクルマーカーにはログのみ

### 3. チャネルを追加する前に重複を折り畳む

探す：
- GitHub、Linear、ローカルログに現れる同じ PR イベント
- 同じ失敗に対する繰り返しフック通知
- 生で転送される代わりに要約されるべきコメントまたはステータスチャーン
- より良いアクションパスを追加せずに互いを重複させるチャネル

推奨：
- 1 つの正規サマリ
- 1 つの所有者
- 1 つの主要チャネル
- 1 つのフォールバックパス

### 4. ECC ネイティブワークフローを設計

各実通知ニーズについて、定義：
- **source**
- **gate**
- **shape**：即時アラート、ダイジェスト、キュー、ダッシュボードのみ
- **channel**
- **action**

ECC にすでにプリミティブがある場合、推奨：
- オペレータトリアージ用スキル
- 自動発出／強制用フック
- 委任分類用エージェント
- 実ブリッジが欠けているときのみ MCP／コネクタ

### 5. アクションバイアスのある設計を返す

以下で終わる：
- 何を保持
- 何を抑制
- 何をマージ
- 次に ECC が何をラップすべきか

## 出力フォーマット

```text
CURRENT SURFACE
- sources
- channels
- duplicates
- gaps

EVENT MODEL
- critical
- high
- medium
- low

ROUTING PLAN
- source -> channel
- why
- operator owner

CONSOLIDATION
- suppress
- merge
- canonical summaries

NEXT ECC MOVE
- skill / hook / agent / MCP
- exact workflow to build next
```

## 推奨ルール

- 多くの弱いレーンより 1 つの強いレーンを推奨
- 中および低シグナル更新にはダイジェストを推奨
- シグナルが自動的に発出されるべきときはフックを推奨
- 作業がトリアージ、ルーティング、レビュー優先意思決定のときにはオペレータスキルを推奨
- 根本原因がアラートではなくバックログ／PR 調整のときには `project-flow-ops` を推奨
- ユーザーが最初にソースインベントリを必要とするときには `workspace-surface-audit` を推奨
- デスクトップ通知で十分なら、不要な外部ブリッジを発明しない

## 良いユースケース

- 「GitHub、Linear、ローカルフックアラートがあるが、単一のオペレータフローがない」
- 「CI 失敗がノイズが多く、人々が無視する」
- 「Claude、OpenCode、Codex サーフェス全体で 1 つの通知ポリシーが欲しい」
- 「何が中断対ダイジェストに落ちるかを判断して」
- 「重複する通知 PR アイデアを 1 つの正規 ECC レーンに折り畳む」
