# ECC Tools コスト監査

ECC Tools の GitHub App がコスト消費過剰、PR を過剰作成、利用上限をバイパス、無料ユーザーをプレミアム解析パスへルーティングしている可能性をユーザーが疑う場合に用いる。

これは隣接の [ECC-Tools](../../ECC-Tools) リポジトリ向けのフォーカスされたオペレータワークフローである。汎用課金スキルでもなく、リポジトリ全体のコードレビュー走査でもない。

## スキルスタック

関連時に以下の ECC ネイティブスキルをワークフローへ取り込む。

- `autonomous-loops`: webhook・キュー・課金・リトライをまたぐ境界付きマルチステップ監査
- `agentic-engineering`: リクエストパスを離散的・証明可能な単位に分解する
- `customer-billing-ops`: リポジトリ挙動と顧客影響計算を明確に切り分ける必要があるとき
- `search-first`: ヘルパー再発明やリポジトリ内ユーティリティ再実装を避ける
- `security-review`: 認証・利用ゲート・エンタイトルメント・シークレットに触れるとき
- `verification-loop`: rerun 安全性と修正後の正確な状態の証明
- `tdd-workflow`: ワーカー・ルーター・課金経路の修正に回帰カバレッジが必要なとき

## 利用タイミング

- ユーザーが ECC Tools のバーンレート、PR 再帰、PR 過剰作成、利用上限バイパス、プレミアムモデル漏洩に言及
- タスクが隣接 `ECC-Tools` リポジトリ内で、webhook ハンドラ・キューワーカー・利用予約・PR 作成ロジック・有料ゲート強制に依存する
- カスタマーレポートが「アプリが PR を作りすぎた」「課金が誤っている」「使えない結果のままコードを解析した」と述べる

## スコープのガードレール

- 隣接 `ECC-Tools` リポジトリで作業し、`everything-claude-code` では作業しない
- ユーザーが明確に修正を求めない限り、まず read-only で開始する
- 解析バーンを追跡中に、無関係の billing・checkout・UI フローを書き換えない
- アプリ生成ブランチおよびアプリ生成 PR は、反証されるまで再帰経路の赤旗として扱う
- 以下3つを明示的に分離する:
  - リポジトリ側のバーン原因
  - 顧客向け課金影響
  - バックログ追跡が必要なプロダクトまたはエンタイトルメントの欠如

## ワークフロー

### 1. リポジトリスコープを固定する

- 隣接 `ECC-Tools` リポジトリに切り替える
- まずブランチとローカル diff を確認する
- 監査対象の正確な表面を特定する:
  - webhook ルーター
  - キュープロデューサー
  - キューコンシューマー
  - PR 作成経路
  - 利用予約 / 課金経路
  - モデルルーティング経路

### 2. 仮説より先にイングレスを追跡する

- まず `src/index.*` または主エントリポイントを確認する
- 修正提案前にすべての enqueue 経路をマップする
- どの GitHub イベントが同一キュータイプを共有するか確認する
- push、pull_request、synchronize、comment、手動再実行イベントが同一の高コスト経路に収束しうるかを確認する

### 3. ワーカーと副作用を追跡する

- 解析を扱うキューコンシューマーまたはスケジュールワーカーを確認する
- キュー解析が常に以下で終わるかを確認する:
  - PR 作成
  - ブランチ作成
  - ファイル更新
  - プレミアムモデル呼び出し
  - 利用量インクリメント
- 解析がトークンを消費後、出力永続化前に失敗できるなら、それを burn-with-broken-output として分類する

### 4. 高シグナルなバーン経路を監査する

#### PR 増殖

- PR ヘルパーとブランチ命名を確認する
- 重複排除・synchronize イベント処理・既存 PR 再利用を確認する
- アプリ生成ブランチが解析に再進入できる場合、それを priority-0 の再帰リスクとして扱う

#### クォータバイパス

- クォータがチェックされる箇所と、利用が予約・インクリメントされる箇所を確認する
- クォータが enqueue 前にチェックされるが課金がワーカー内のみで行われるなら、並行フロントドア通過を実レースとして扱う

#### プレミアムモデル漏洩

- モデル選定・ティア分岐・プロバイダルーティングを確認する
- プレミアムキーが存在するときに無料/上限ユーザーがプレミアム解析器に到達できるかを検証する

#### リトライバーン

- リトライループ・重複キュージョブ・決定論的失敗の再実行を確認する
- 同一の非一過性エラーで解析が繰り返し消費されうるなら、品質改善より先にそれを直す

### 5. バーン順に修正する

ユーザーがコード変更を求めた場合、以下の順序で優先する。

1. 自動 PR 増殖の停止
2. クォータバイパスの停止
3. プレミアム漏洩の停止
4. 重複ジョブのファンアウトと意味のないリトライの停止
5. rerun/update 安全性ギャップの閉鎖

同一原因が明確に複数ファイルに渡る場合を除き、1〜3件の直接修正に限定する。

### 6. 最小の証明ステップで検証する

- 変更経路をカバーする対象テストまたは統合スライスのみを再実行する
- バーン経路が以下のいずれかになっているか検証する:
  - ブロック済み
  - 重複排除済み
  - より安価な解析にダウングレード
  - または早期拒否
- 最終状態を厳密に記述する:
  - ローカル変更済み
  - ローカル検証済み
  - push 済み
  - デプロイ済み
  - まだブロック中

## 高シグナルな障害パターン

### 1. すべてのトリガに対して同一キュータイプ

push、PR 同期、手動監査すべてが同じジョブを enqueue し、ワーカーが必ず PR を作るなら、解析 = PR スパムである。

### 2. enqueue 後の利用予約

利用がフロントドアでチェックされ、ワーカー内でのみインクリメントされるなら、並行リクエストがすべてゲートを通過してクォータを超過しうる。

### 3. プレミアム経路への無料ティア

無料キュージョブが、キー存在時に依然として Anthropic や他のプレミアムプロバイダにルーティングされるなら、ユーザーがプレミアム結果を見ない場合でも実コスト漏洩である。

### 4. アプリ生成ブランチが webhook に再進入

`pull_request.synchronize`、ブランチ push、コメント起動の実行がアプリ所有ブランチで発火するなら、アプリは自身の出力を再帰的に解析しうる。

### 5. 永続化安全性前の高コスト処理

トークンを消費した後に PR 作成・ファイル更新・ブランチ衝突で失敗できるなら、価値を出さずにコストを燃やしている。

## 落とし穴

- 広範なリポジトリ徘徊から始めない。まず webhook → キュー → ワーカーを固定する
- 顧客課金推論とコード裏付けのプロダクト真実を混ぜない
- 最高バーン経路を封じる前により価値の低い品質課題を直さない
- 狭い証明ステップを再実行するまで「バーン修正済み」と主張しない
- ユーザーが求めない限り push やデプロイしない
- 進行中の無関係なローカル変更には触れない

## 検証

- 原因は正確なファイルパスとコード領域を引用する
- 修正はコード美観ではなくバーン影響で順序付けする
- 証明コマンドが名指しされている
- 最終ステータスがローカル変更・検証・push・デプロイを区別する
