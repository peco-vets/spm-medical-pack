# インターフェースをより良い感触にする

より洗練されたインターフェースに積み重なる小さなデザインエンジニアリングのディテールには、このスキルを使う。

ソース: `linus707` による休眠コミュニティ PR #1659 から救出。

## 使用するタイミング

- ユーザーが UI がしっくりこない、平坦、汎用的、窮屈、ジャンプする、未完成と発言した場合
- コントロール、カード、リスト、ダッシュボード、ナビゲーション、フォーム、ツールバーを構築している場合
- コンポーネントが hover、active、focus、enter、exit、loading、または empty 状態を必要とする場合
- フロントエンドレビューが具体的な before/after 推奨事項を必要とする場合

## 基本原則

### 同心円の半径

近接するネストされた角丸サーフェスに対して:

```text
outer radius = inner radius + padding
```

パディングが大きい場合、数式に固執するのではなく層を別個のサーフェスとして扱う。ポイントは光学的一貫性であり、数式崇拝ではない。

### 光学的アライメント

幾何学的中央寄せが必ずしも視覚的中央寄せではない。アイコンボタン、再生三角形、矢印、星、非対称アイコンは小さなオフセットを必要とすることが多い。可能であれば SVG を修正する。それ以外はピクセルレベルのマージンまたはパディング変更で調整する。

### シャドウと境界線

分離とフォーカスリングには境界線を使う。カード、ボタン、ドロップダウン、ポップオーバーに深さが必要な場合は層化されたシャドウを使う。シャドウは透明で、背景を越えて機能するように十分繊細であるべきである。

### テキスト折り返し

- 見出しと短いタイトルには `text-wrap: balance` を使う
- 短から中程度の本文、キャプション、説明、リストアイテムには `text-wrap: pretty` を使う
- 長い散文、コード、preformatted コンテンツには両方を避ける
- カウンター、タイマー、価格、テーブル、その他の更新数値には `font-variant-numeric: tabular-nums` を使う

### フォントスムージング

macOS では、プロジェクトがまだ行っていない場合、ルートレイアウトでアンチエイリアスフォントスムージングを適用する:

```css
html {
  -webkit-font-smoothing: antialiased;
  -moz-osx-font-smoothing: grayscale;
}
```

### 画像のアウトライン

画像はサーフェスに端がぼやけないように、しばしば繊細な内側アウトラインを必要とする。

```css
img {
  outline: 1px solid rgba(0, 0, 0, 0.1);
  outline-offset: -1px;
}

@media (prefers-color-scheme: dark) {
  img {
    outline-color: rgba(255, 255, 255, 0.1);
  }
}
```

中立の黒または白アルファアウトラインを使う。ブランドパレットで画像アウトラインに色を付けない。

### モーション

ユーザーがモーション途中で意図を変えたときに再ターゲットできるため、インタラクティブな状態変化には CSS トランジションを使う。ステージ化されたワンショットの入場やローディングシーケンスのために keyframes を予約する。

良いモーションのデフォルト:

- Enter: opacity、小さな `translateY`、オプションでブラーを組み合わせる
- Exit: enter より短く静か、通常 150ms
- Press: 触覚的ボタンには `scale(0.96)`、動きが気を散らすときに無効化する方法を用意
- アイコンスワップ: 即座の visibility トグルの代わりに、opacity、scale、blur でクロスフェード

### トランジションスコープ

`transition: all` を絶対に使わない。変更されるプロパティを指定する:

```css
.button {
  transition-property: transform, background-color, box-shadow;
  transition-duration: 150ms;
  transition-timing-function: ease-out;
}
```

`will-change` は `transform`、`opacity`、`filter` などコンポジターフレンドリーなプロパティの初フレームのスタッターにのみ使う。`will-change: all` は絶対に使わない。

### ヒットエリア

インタラクティブなコントロールは少なくとも 40x40px のヒットエリアを持つべきで、レイアウトが許せば理想的には 44x44px。可視アイコンが小さい場合は擬似要素で拡張するが、拡張されたヒットエリアを重ねさせない。

## レビュー出力

UI ポリッシュパスをレビューする際は、before/after 行で具体的な変更を報告する:

| 原則 | Before | After |
| --- | --- | --- |
| 同心円の半径 | 親と子で同じ半径 | 親の半径がパディングを考慮 |
| Tabular numbers | カウンターが桁変更でシフト | カウンターが `tabular-nums` を使用 |
| トランジションスコープ | `transition: all` | 明示的なトランジションプロパティ |

スニペットから明らかでない場合はファイルパスとプロパティを含める。チェックしたが変更しなかった原則は省略する。

## チェックリスト

- ネストされた角丸要素が光学的に一貫している
- アイコンが視覚的に中央寄せされている
- ボタン、カード、ポップオーバーが適切な理由で境界線またはシャドウを使う
- 見出しと短いテキストが不格好な折り返しを避けている
- 動的な数値が tabular numerals を使う
- 必要な場所で画像が中立のアウトラインを持つ
- 入場と退場のアニメーションが分かれ、繊細で、適切な場合は中断可能
- ボタンに誇張されたモーションのない触覚的アクティブ状態がある
- `transition: all` と `will-change: all` が存在しない
- 小さなコントロールでも使用可能なヒットエリアがある
