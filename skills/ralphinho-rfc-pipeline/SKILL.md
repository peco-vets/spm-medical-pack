# Ralphinho RFC パイプライン

[humanplane](https://github.com/humanplane) スタイルの RFC 分解パターンとマルチユニットオーケストレーションワークフローに着想を得ている。

機能が単一エージェントパスには大きすぎ、独立に検証可能なワークユニットに分割する必要がある場合に、このスキルを使う。

## パイプラインステージ

1. RFC 取り込み
2. DAG 分解
3. ユニット割り当て
4. ユニット実装
5. ユニット検証
6. マージキューと統合
7. 最終システム検証

## ユニット仕様テンプレート

各ワークユニットには以下を含める：
- `id`
- `depends_on`
- `scope`
- `acceptance_tests`
- `risk_level`
- `rollback_plan`

## 複雑度ティア

- ティア 1：孤立したファイル編集、決定的テスト
- ティア 2：複数ファイルの動作変更、中程度の統合リスク
- ティア 3：スキーマ／認証／パフォーマンス／セキュリティ変更

## ユニットごとの品質パイプライン

1. 調査（research）
2. 実装計画
3. 実装
4. テスト
5. レビュー
6. マージ準備完了レポート

## マージキュールール

- 未解決の依存関係失敗があるユニットは決してマージしない
- ユニットブランチは常に最新の統合ブランチに対してリベースする
- 各キューマージ後に統合テストを再実行する

## リカバリ

ユニットが停滞した場合：
- アクティブキューから除外する
- 知見をスナップショットする
- スコープを狭めたユニットを再生成する
- 更新された制約で再試行する

## 出力

- RFC 実行ログ
- ユニットスコアカード
- 依存関係グラフスナップショット
- 統合リスクサマリー
