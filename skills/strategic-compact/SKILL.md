# Strategic Compact スキル

任意の自動コンパクションに頼るのではなく、ワークフローの戦略的ポイントで手動 `/compact` を提案する。

## 起動するタイミング

- コンテキスト制限（200K+ トークン）に近づく長いセッションを実行している
- 複数フェーズタスク（調査 → 計画 → 実装 → テスト）を作業している
- 同じセッション内で関連のないタスク間で切り替える
- 主要マイルストーン完了後、新しい作業を開始する
- レスポンスが遅くなる、または首尾一貫しなくなる（コンテキストプレッシャー）

## なぜ戦略的コンパクションか？

自動コンパクションは任意のポイントでトリガーされる：
- しばしばタスクの途中で重要なコンテキストを失う
- 論理的なタスク境界の認識なし
- 複雑な複数ステップ操作を中断する可能性がある

論理境界での戦略的コンパクション：
- **探索後、実行前** — リサーチコンテキストをコンパクト、実装計画を保持
- **マイルストーン完了後** — 次のフェーズへの新鮮な開始
- **主要なコンテキスト切り替え前** — 異なるタスクの前に探索コンテキストをクリア

## 動作の仕組み

`suggest-compact.js` スクリプトは PreToolUse（Edit/Write）で実行され：

1. **ツール呼び出しを追跡** — セッション内のツール呼び出しをカウント
2. **閾値検出** — 設定可能な閾値（デフォルト：50 呼び出し）で提案
3. **定期的リマインダ** — 閾値後 25 呼び出しごとにリマインド

## フックセットアップ

`~/.claude/settings.json` に追加：

```json
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Edit",
        "hooks": [{ "type": "command", "command": "node ~/.claude/scripts/hooks/suggest-compact.js" }]
      },
      {
        "matcher": "Write",
        "hooks": [{ "type": "command", "command": "node ~/.claude/scripts/hooks/suggest-compact.js" }]
      }
    ]
  }
}
```

## 設定

環境変数：
- `COMPACT_THRESHOLD` — 最初の提案までのツール呼び出し（デフォルト：50）

## コンパクション意思決定ガイド

いつコンパクトするかを決めるためにこのテーブルを使う：

| フェーズ遷移 | コンパクト？ | 理由 |
|-----------------|----------|-----|
| 調査 → 計画 | Yes | 調査コンテキストはかさばる、計画は蒸留された出力 |
| 計画 → 実装 | Yes | 計画は TodoWrite またはファイル内、コードのためにコンテキストを解放 |
| 実装 → テスト | Maybe | テストが最近のコードを参照するなら保持、焦点を切り替えるならコンパクト |
| デバッグ → 次の機能 | Yes | デバッグトレースが無関係な作業のコンテキストを汚染 |
| 実装途中 | No | 変数名、ファイルパス、部分状態を失うのはコストが高い |
| 失敗したアプローチ後 | Yes | 新しいアプローチを試す前に行き止まりの推論をクリア |

## コンパクションを生き残るもの

何が持続するかを理解すると自信を持ってコンパクトできる：

| 持続する | 失われる |
|----------|------|
| CLAUDE.md 指示 | 中間推論と解析 |
| TodoWrite タスクリスト | 以前読んだファイル内容 |
| メモリファイル（`~/.claude/memory/`） | 複数ステップ会話コンテキスト |
| Git 状態（コミット、ブランチ） | ツール呼び出し履歴とカウント |
| ディスク上のファイル | 口頭で述べられたニュアンスのあるユーザー設定 |

## ベストプラクティス

1. **計画後にコンパクト** — TodoWrite で計画が最終化されたら、新鮮に開始するためにコンパクト
2. **デバッグ後にコンパクト** — 続行前にエラー解決コンテキストをクリア
3. **実装途中にコンパクトしない** — 関連変更のためにコンテキストを保持
4. **提案を読む** — フックは*いつ*を教え、あなたが*かどうか*を決める
5. **コンパクト前に書き出す** — コンパクト前に重要なコンテキストをファイルまたはメモリに保存
6. **サマリ付きで `/compact` を使う** — カスタムメッセージを追加：`/compact Focus on implementing auth middleware next`

## トークン最適化パターン

### トリガーテーブル遅延ロード
セッション開始時にフルスキルコンテンツをロードする代わりに、キーワードをスキルパスにマップするトリガーテーブルを使う。スキルはトリガーされたときのみロードされ、ベースラインコンテキストを 50%+ 削減する：

| トリガー | スキル | ロードタイミング |
|---------|-------|-----------|
| "test", "tdd", "coverage" | tdd-workflow | ユーザーがテストに言及 |
| "security", "auth", "xss" | security-review | セキュリティ関連の作業 |
| "deploy", "ci/cd" | deployment-patterns | デプロイコンテキスト |

### コンテキスト構成の認識
コンテキストウィンドウを消費しているものを監視：
- **CLAUDE.md ファイル** — 常にロード、簡潔に保つ
- **ロードされたスキル** — 各スキルが 1-5K トークンを追加
- **会話履歴** — 各交換で成長
- **ツール結果** — ファイル読み取り、検索結果がかさを追加

### 重複指示検出
重複コンテキストの一般的なソース：
- `~/.claude/rules/` とプロジェクト `.claude/rules/` の両方の同じルール
- CLAUDE.md 指示を繰り返すスキル
- 重複ドメインをカバーする複数のスキル

### コンテキスト最適化ツール
- `token-optimizer` MCP — コンテンツ重複排除による自動 95%+ トークン削減
- `context-mode` — コンテキスト仮想化（315KB から 5.4KB を実証）
