# Blender モーションステート検査

## 利用するタイミング

- Blender キャラクターがアニメーション中にねじれ、ミラー、平坦化、オフセット、または足滑りしているように見える。
- ユーザーがインポートされたアバター、アーマチュア、またはリターゲットされたモーションが期待されるポーズに一致するか尋ねる。
- レンダリングされた証拠を、ボーン、バウンディングボックス、接触、向きベクトルといった構造化された事実と比較する必要がある。
- ワークフローがモデルがキャラクター、プロップ、プロキシメッシュ、コントロールリグ、または壊れたインポートのどれかを判断することに依存する。

## 中核原則

アニメーションされた 3D アセットをスクリーンショットだけで判断しないこと。スクリーンショットはレビュー証拠だが、軸規約、ボーン名、オブジェクトスケール、ローカル変換、ペアレント化されたメッシュ、マテリアルスロット、フレームごとの接触状態を隠す。

まず構造化された Blender 状態を抽出し、それからビューポートスクリーンショットやレンダーを使って事実が示唆することを確認する。

## 仕組み

1. モーションを判断する前に、クリーンなシーンとアセットのベースラインを確立する。
2. エクスポーターまたは Blender 自身のインタープリタ内で実行する Blender Python を使って Blender から構造化された事実を抽出する。
3. 接触、向き、スケール、リターゲティングエラーを最も露出しやすいフレームをサンプリングする。
4. 測定された事実を、ユーザーが期待するポーズ、方向、地面、レンダーゴールと比較する。
5. 確認された事実、考えられる原因、必要な修正を分離した簡潔なレポートを返す。

## 検査ワークフロー

1. シーンをインベントリする。
   - メッシュ、アーマチュア、エンプティ、カメラ、ライト、モディファイヤ、ペアレント関係、非表示オブジェクトをリストする。
   - アバターを判断する前にキャラクターメッシュをヘルパー/プロキシジオメトリと分離する。
   - オブジェクト空間とワールド空間のバウンディングボックスを記録する。

2. スケルトンを特定する。
   - アーマチュア名、ポーズボーン、ボーンヘッド/テール、ロール、親チェーン、コンストレイント、レストポーズ軸をキャプチャする。
   - 腰・脊椎・首・頭・肩・肘・手・腿・膝・足首・足のようなセマンティックなボーンをマップする。
   - 欠落した左右ペアと珍しい命名スキームをフラグする。

3. forward・up・side 軸を判断する。
   - 骨盤・脊椎・肩・腰・頭・足を一緒に使う。単一のメッシュ法線に頼らない。
   - ローカルアーマチュア軸をワールド軸と glTF Y-up vs Blender Z-up のようなインポートファイル規約と比較する。
   - 顔/頭/足の方向がルートモーションと矛盾するとき、ミラー化または反対方向のインポートをマークする。

4. アニメーションフレームをサンプリングする。
   - 最初、中間、接触、空中、極端なフレームを検査する。
   - ルート位置、ルートの向き、骨盤の高さ、胴体の傾き、四肢の方向、足のクリアランス、メッシュ境界を記録する。
   - 長いまたは速いモーションでは、フリップ、着地、ターン、衝突、床接触の周辺をより密にサンプリングする。

5. リターゲティングの責任にする前にモデル整合性を確認する。
   - アニメーション適用前のクリーンベースライン形状を確認する。
   - ユーザーが明示的に修復を求めない限り、元のメッシュ、マテリアル、アーマチュア、スキニングを保持する。
   - 不可解な球状の塊、巨大なプロキシメッシュ、または潰れた体を、それ以外が証明されるまでインポート/選択の問題として扱う。

6. 接触とモーションの問題を診断する。
   - 地面貫通: 各フレームで最も低い足または靴の頂点を床の高さと比較する。
   - 足滑り: 接地フレーム間で足のワールド位置を比較する。
   - 脚の交差: 左右の腿、膝、足首、足のサイド順序を比較する。
   - ねじれダメージ: 肢軸周りのロール/ツイストとは別にボーンのスウィング方向を比較する。
   - スケールドリフト: アニメーションされたメッシュ境界をクリーンベースラインの境界と比較する。

7. 意見の前に事実を報告する。
   - フレーム番号、オブジェクト名、ボーン名、ワールド座標、しきい値を含める。
   - 確認された失敗を視覚的疑いから分離する。
   - 構造化された状態が何を見るべきかを説明した後にのみスクリーンショットを添付する。

## 例

### 足滑りのある歩行サイクル

シナリオ: リターゲットされたキャラクターが歩行サイクル中にスケートしているように見えるが、前面カメラ角度では足の接触を判断しにくい。

ワークフローを適用:
- シーンをインベントリ: キャラクターメッシュ `HeroBody`、アーマチュア `HeroRig`、地面プレーン `Floor`、隠れたプロキシメッシュなし。
- スケルトンを特定: セマンティックな足は `foot.L` と `foot.R`、腰は `pelvis`、ルートボーンは `root`。
- アニメーションフレームをサンプル: 接地フレーム付近の 1, 18, 24, 30, 42, 48 フレームを検査。
- 接触とモーションの問題を診断: 接地フレーム中の足ワールド位置を比較する。

抽出された事実:

| Frame | Fact | Evidence |
| --- | --- | --- |
| 18 | Left foot is planted | `foot.L min_z = 0.004`, toe and heel both near floor |
| 24 | Left foot slides while planted | `foot.L x = 0.21 -> 0.28` over six frames |
| 30 | Pelvis keeps moving forward | `pelvis y = 1.14 -> 1.31` |

判定: レンダー準備状態として fail。モーションは foot-lock クリーンアップまたはリターゲティングコンストレイントレビューが必要。ボディメッシュはプロポーション変更を必要としない。

### 反対方向にインポートされたキャラクター

シナリオ: キャラクターは静止フレームでは正しく見えるが、アニメーションは期待される進行方向と反対に動く。

ワークフローを適用:
- forward・up・side 軸を判断: 頭、胸、足、ルートモーションを比較する。
- アニメーションフレームをサンプル: フレーム 1 と進行パスの中間点を検査。
- 意見の前に事実を報告: ルートの向きとモデルの向きを別々に含める。

抽出された事実:

| Frame | Fact | Evidence |
| --- | --- | --- |
| 1 | Character face points toward world `-Y` | head/chest vector from `neck` to `head` resolves to `-Y` |
| 72 | Root motion travels toward world `+Y` | `root y = 0.0 -> 2.8` |
| 72 | Feet remain visually forward-facing opposite travel | toe bones point `-Y` while displacement is `+Y` |

判定: 反対方向インポートまたはリターゲティングの forward 軸ミスマッチの可能性が高い。アニメーションカーブを編集する前にインポート/リターゲット軸マッピングを修正する。

## 実用的なしきい値

- シーンユニットスケールが別途指定しない限り Blender のデフォルトメートルスケール単位を仮定する。
- 床が柔らかいか意図的にスタイル化されていない限り、1-2 cm を超える地面貫通は視覚的だと扱う。
- 5% を超える突然のスケール変化はリグ、コンストレイント、または変換継承の問題である可能性が高い。
- 空中の反転モーション中に左右足首のサイド順序フリップが起きる場合、後で回復しても脚の交差リスクとして扱う。
- ソースモーションにスナップターンが含まれない限り、フレーム当たり 30 度を超えるルートの向きのジャンプは怪しいと扱う。

## アンチパターン

- タスクが明示的にメッシュ修復でない限り、ポーズマッチングを強制するためにボディプロポーションを変更しない。
- 記録する前にクリーンベースラインをベイクして消さない。
- レンダリングされた 1 つのカメラ角度をポーズが正しい証拠として使わない。
- ヘルパーオブジェクトがキャラクターの一部でない理由を記録するまで削除しない。
- 頭、足、胴体、ルートモーションを一緒に確認せずに、アバターが +Y、-Y、+X、-X を向いていると仮定しない。

## ツーリングノート

Blender ステートエクスポーターが利用可能なら、メッシュ、アーマチュア、ポーズボーン、マテリアル、接触、バウンディングボックス、サンプリングされたアニメーションフレームを含む JSON を優先する。エクスポーターが存在しないなら、`bpy` は通常のシステム Python インタープリタでは利用できないため、Blender 自身を介して小さな Blender Python スクリプトを実行する (例: `blender --background scene.blend --python collect_motion_state.py`)。
