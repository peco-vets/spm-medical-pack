# Production Audit

ユーザーがアプリケーションが出荷準備ができているか、本番で何が壊れる可能性があるか、またはローンチ前に何を修正する必要があるかを尋ねる場合にこのスキルを使う。これは古いコミュニティ production-audit アイデアのメンテナ安全な書き直しである: 有用な本番準備度レンズを保持し、ピンされていない外部実行とサードパーティデータ共有を削除する。

## 使用するタイミング

- ユーザーが「is this production-ready」、「what would break in prod」、「what did we miss」、「audit this repo」、または「ready to ship?」と尋ねる
- フィーチャーがマージされデプロイ前またはマージ後のリスクパスが必要
- 公開ローンチ、デモ、顧客ロールアウト、または投資家ウォークスルーが近い
- CI はグリーンだが、ユーザーがテスト状態だけでなく本番リスクを求めている
- 証拠収集のためにデプロイされた URL、リリースブランチ、PR、または現在のチェックアウトが利用可能

## 使用しないタイミング

- 適切なレンズが行レベルのセキュアコーディングであるアクティブな実装中。最初に `security-review` を使う
- 純粋なライブラリ、テンプレート、ドキュメントのみのリポ、または足場 (ユーザーがアプリケーション準備度ではなくパッケージング/リリース準備度を求める場合を除く)
- ユーザーが正式なコンプライアンス監査を依頼するとき。このスキルはエンジニアリングトリアージであり、法律、財務、医療、または規制認証ではない
- 利用可能な証拠がリポ、デプロイ、CI、またはランタイムサーフェスのない製品アイデアのみ

## 動作の仕組み

ローカルとユーザー承認の証拠から監査を構築する。ピンされていないリモートコードを実行したり、リポジトリコンテンツをサードパーティサービスにアップロードしたり、ユーザーがその特定のツールとデータフローを明示的に承認しない限り外部スキャナを呼び出したりしない。

この順序を使う:

1. リリースサーフェスを確立する
2. 最近の変更と現在のブランチ状態を読む
3. 実際にリポに存在するランタイム、認証、データ、支払い、バックグラウンドジョブ、AI、デプロイメント境界を検査
4. CI、テスト、マイグレーション、環境ドキュメント、ロールバックパスをチェック
5. 具体的な修正を伴う短いship/block 推奨を生成

## 証拠チェックリスト

安価なローカルシグナルから始める:

```text
git status --short --branch
git log --oneline --decorate -20
git diff --stat origin/main...HEAD
```

その後、プロジェクト固有のサーフェスを検査:

- パッケージスクリプト、CI ワークフロー、リリーススクリプト、Docker ファイル、デプロイメントマニフェスト
- API ルート、Webhook、認証ミドルウェア、バックグラウンドワーカー、cron ジョブ、データベースマイグレーション
- 環境変数ドキュメントと起動チェック
- 観測可能性フック、エラーレポート、ログ、ヘルスチェック、ダッシュボード
- ロールバック、シード、マイグレーション、バックフィル手順
- 最も重要なユーザーパスの E2E カバレッジ

デプロイされた URL がスコープにある場合、その URL に対してのみブラウザまたは HTTP チェックを使い、ユーザーが安全なテストアカウントを提供しない限り認証されたアクションを避ける。

## リスクレンズ

### セキュリティと認証

- パブリックルート、API ルート、管理者ルートは明確に分離されているか?
- 認証と認可はサーバー側で強制されるか?
- シークレットはクライアントバンドル、ログ、サンプル出力、チェックインされたファイルから保たれるか?
- アプリが必要とする場所でレート制限、CSRF 保護、CORS ポリシー、アップロード検証は存在するか?
- AI またはエージェントサーフェスはプロンプトインジェクション、ツール乱用、特権アクションへの信頼されないコンテンツのクロスから防御するか?

### データ整合性

- マイグレーションはクリーンに前方実行され、ロールバックまたは回復計画があるか?
- 破壊的マイグレーション、バックフィル、データインポートは安全にステージされるか?
- データベースポリシー、付与、サービスロール境界はアプリのテナンシーモデルと一致するか?
- 書き込み、ジョブ、Webhook ハンドラのリトライは冪等か?

### 支払いと Webhook

- Webhook シグネチャは信頼されたペイロードフィールドをパースする前に検証されるか?
- 各支払い、サブスクリプション、または履行 Webhook は冪等か?
- 再生、重複配送、順不同配送は処理されるか?
- テストモードとライブモードの認証情報は分離されているか?

### 運用

- アプリは文書化されたコマンドを使ってクリーンチェックアウトから起動できるか?
- 必要な環境変数は命名され、検証され、フェイルファストか?
- 依存関係が到達可能であることを証明するヘルスチェックがあるか?
- デプロイ、ロールバック、インシデントオーナーパスは文書化されているか?
- ログはシークレットや個人データをリークせずに有用か?

### ユーザー体験

- ローンチクリティカルパスはデスクトップとモバイルでカバーされるか?
- フォームは入力ズーム、レイアウトオーバーラップ、ブロックされた送信状態なしでモバイルで使用可能か?
- ローディング、空、エラー、許可拒否状態はユーザーに何が起こったか伝えるか?
- クリティカル操作が失敗したときのサポートまたは回復パスはあるか?

## スコアリング

スコアを使って優先順位付けを強制する。数学的確実性を含意しない。

| バンド | スコア | 意味 |
| --- | --- | --- |
| Blocked | 0-49 | トップリスクが修正されるまで出荷しない |
| Risky | 50-69 | 小さなロールアウトや内部ベータの背後でのみ出荷 |
| Launchable With Caveats | 70-84 | オーナーがリストされたリスクを受け入れる場合に出荷 |
| Strong | 85-100 | 利用可能な証拠から明らかなローンチブロッカーなし |

これらのいずれかが真の場合、スコアを `69` に上限:

- 機密データに認証または認可が欠落している
- 支払いまたは履行 Webhook が冪等でない
- 必要なマイグレーションが安全に実行できない
- シークレットがクライアントバンドル、ログ、またはコミットされたファイルに露出している
- 高インパクトリリースのロールバックパスがない

CI がグリーンでないか、ローンチクリティカルパスがエンドツーエンドでテストされていない場合、スコアを `84` に上限。

## 出力形式

1 文でリード:

```text
Production audit: 76/100, launchable with caveats, with webhook idempotency and rollback docs as the two risks to fix before public launch.
```

その後リスト:

- `Blockers`: デプロイ前の必修正アイテム
- `High-value fixes`: ユーザーがスコアを改善したい場合の次の修正
- `Evidence checked`: 検査されたファイル、コマンド、CI、デプロイされた URL、または PR
- `Evidence missing`: 提供されると信頼度を変える可能性のあるもの
- `Next action`: 1 つの具体的な修正または検証ステップ

強みは短く保つ。ユーザーは準備度を求めたので、有用な答えは残りのリスクと次のアクションである。

## 例

ユーザー:

```text
is this ready to ship?
```

レスポンス:

```text
Production audit: 68/100, risky, because Stripe webhooks are verified but not idempotent and there is no rollback note for the pending migration.

Blockers:
- Add idempotency for `checkout.session.completed` before fulfilling orders.
- Write and test the rollback path for `20260511_add_billing_state.sql`.

High-value fixes:
- Add a health check that verifies database and payment-provider reachability.
- Add one E2E path for upgrade, webhook fulfillment, and billing-page refresh.

Evidence checked:
- `api/stripe/webhook.ts`
- `db/migrations/20260511_add_billing_state.sql`
- GitHub Actions run for the release branch

Next action: Want me to patch webhook idempotency first?
```

## アンチパターン

- デフォルト監査パスとして `npx <package>@latest` やリモートスキャナを実行
- 明示的承認なしにソース、シークレット、顧客データ、またはプライベートトポロジを外部監査サービスにアップロード
- 検査された証拠を命名せずにスコアを生成
- グリーン CI を本番準備度として扱う
- 汎用的な「let me know what you want to do」で終わる
