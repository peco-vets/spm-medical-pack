# フロントエンドスライド

ブラウザ単体で動作する、ゼロ依存・アニメーション豊富な HTML プレゼンテーションを作成する。

zarazhangrui（クレジット: @zarazhangrui）の作品で示された視覚的探索アプローチに着想を得ている。

## 起動タイミング

- トークデック・ピッチデック・ワークショップデック・社内プレゼンの作成
- `.ppt` または `.pptx` から HTML プレゼンへの変換
- 既存 HTML プレゼンのレイアウト・モーション・タイポグラフィの改善
- デザイン嗜好が未確定のユーザーとプレゼンスタイルを探索

## 譲れない要件

1. **ゼロ依存**: デフォルトはインライン CSS と JS を含む単一の自己完結 HTML ファイル
2. **viewport fit が必須**: 全スライドが1ビューポート内に収まり内部スクロールしない
3. **show, don't tell**: 抽象的なスタイルアンケートではなく視覚プレビューを使う
4. **特徴的なデザイン**: 紫グラデ・白上 Inter・テンプレ感のあるデッキを避ける
5. **本番品質**: コードはコメント・アクセシブル・レスポンシブ・高性能に保つ

生成前に `STYLE_PRESETS.md` を読み、viewport-safe な CSS 基盤、密度上限、プリセットカタログ、CSS の落とし穴を把握する。

## ワークフロー

### 1. モード検出

以下から1つを選ぶ。
- **新規プレゼン**: ユーザーがトピック・ノート・完成稿を持つ
- **PPT 変換**: ユーザーが `.ppt` または `.pptx` を持つ
- **改善**: ユーザーが既存 HTML スライドを持ち、改善を求める

### 2. コンテンツ把握

最小限のみ確認する。
- 目的: ピッチ・教育・カンファレンストーク・社内更新
- 長さ: short (5-10)・medium (10-20)・long (20+)
- コンテンツ状態: 完成コピー・ラフノート・トピックのみ

ユーザーがコンテンツを持つなら、スタイリング前に貼り付けを依頼する。

### 3. スタイル探索

デフォルトは視覚探索である。

ユーザーが希望プリセットを既知なら、プレビューをスキップして直接使う。

そうでない場合:
1. デッキが生むべき感情を確認する: impressed・energized・focused・inspired
2. `.ecc-design/slide-previews/` に**3つの単一スライドプレビューファイル**を生成する
3. 各プレビューは自己完結し、タイポグラフィ/色/モーションを明確に示し、スライドコンテンツがおよそ100行未満であること
4. どのプレビューを採用するか、または要素をどう混ぜるかをユーザーに確認する

ムードをスタイルへマッピングする際は `STYLE_PRESETS.md` のプリセットガイドを使う。

### 4. プレゼン構築

出力は以下のいずれか。
- `presentation.html`
- `[presentation-name].html`

抽出画像やユーザー提供画像を含む場合のみ `assets/` フォルダを使う。

必須構造:
- セマンティックなスライドセクション
- `STYLE_PRESETS.md` の viewport-safe CSS 基盤
- テーマ値用 CSS カスタムプロパティ
- キーボード・ホイール・タッチナビゲーション用 presentation controller クラス
- reveal アニメーション用 Intersection Observer
- reduced-motion 対応

### 5. viewport fit を強制する

これはハードゲートとして扱う。

ルール:
- 各 `.slide` は `height: 100vh; height: 100dvh; overflow: hidden;` を使う
- すべての type と spacing は `clamp()` でスケールする
- コンテンツが収まらない場合は複数スライドに分割する
- 可読サイズを下回るテキスト縮小でオーバーフローを解決しない
- スライド内スクロールバーを許可しない

`STYLE_PRESETS.md` の密度上限と必須 CSS ブロックを用いる。

### 6. 検証

以下サイズで完成デッキを確認する:
- 1920x1080
- 1280x720
- 768x1024
- 375x667
- 667x375

ブラウザ自動化が利用可能なら、スライドがオーバーフローしないこととキーボードナビゲーションが動作することを検証する。

### 7. 引き渡し

引き渡し時:
- ユーザーが保持を望まない限り、一時プレビューファイルを削除する
- 有用ならプラットフォーム適応 opener でデッキを開く
- ファイルパス・使用プリセット・スライド数・容易なテーマカスタマイズ箇所を要約する

現 OS に合った opener を使う:
- macOS: `open file.html`
- Linux: `xdg-open file.html`
- Windows: `start "" file.html`

## PPT / PPTX 変換

PowerPoint 変換:
1. `python3` + `python-pptx` でテキスト・画像・ノートを抽出するのを優先する
2. `python-pptx` が無い場合、インストール許可を確認するか、手動/エクスポートベースのフォールバックワークフローを使う
3. スライド順・スピーカーノート・抽出アセットを保持する
4. 抽出後、新規プレゼンと同じスタイル選択ワークフローを実行する

変換はクロスプラットフォームに保つ。Python で済む作業に macOS 限定ツールへ依存しない。

## 実装要件

### HTML / CSS

- ユーザーが明示的に複数ファイルプロジェクトを望まない限り、インライン CSS と JS を使う
- フォントは Google Fonts または Fontshare から取得可能
- 雰囲気ある背景・強いタイプ階層・明確なビジュアル方向を優先する
- イラストよりも抽象シェイプ・グラデ・グリッド・ノイズ・幾何学を使う

### JavaScript

含めるもの:
- キーボードナビゲーション
- タッチ / スワイプナビゲーション
- マウスホイールナビゲーション
- プログレスインジケータまたはスライド インデックス
- enter 時 reveal アニメーショントリガ

### アクセシビリティ

- セマンティック構造（`main`、`section`、`nav`）を使う
- 可読コントラストを保つ
- キーボードのみのナビゲーションをサポートする
- `prefers-reduced-motion` を尊重する

## コンテンツ密度上限

ユーザーが明示的に密度を要求し可読性が保たれる場合を除き、以下上限を用いる。

| スライドタイプ | 上限 |
|------------|-------|
| Title | 1 heading + 1 subtitle + 任意 tagline |
| Content | 1 heading + 4-6 bullets または 2 short paragraphs |
| Feature grid | 最大6 cards |
| Code | 最大8-10行 |
| Quote | 1 quote + attribution |
| Image | viewport で制約された 1 image |

## アンチパターン

- ビジュアルアイデンティティのない汎用スタートアップグラデーション
- 意図的エディトリアル以外でのシステムフォントデッキ
- 長い箇条書きの壁
- スクロールが必要なコードブロック
- 短画面で崩れる固定高コンテンツボックス
- `-clamp(...)` のような無効な負の CSS 関数

## 納品物チェックリスト

- プレゼンがブラウザでローカルファイルから動作する
- 全スライドがスクロールなしで viewport に収まる
- スタイルが特徴的かつ意図的である
- アニメーションがノイズではなく意味を持つ
- reduced motion が尊重される
- ファイルパスとカスタマイズ箇所が引き渡し時に説明されている
