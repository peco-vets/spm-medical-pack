# Automation Audit Ops

ユーザーが何の自動化が稼働中か、どのジョブが壊れているか、どこに重複があるか、どのツーリングとコネクタが今実際に有用な仕事をしているかを尋ねるときに使う。

これは監査ファーストのオペレーター用スキルである。仕事は、何かを書き直す前に、証拠に裏付けられたインベントリと keep / merge / cut / fix-next の推奨セットを生成することである。

## スキルスタック

関連する場合、これらの ECC ネイティブスキルをワークフローに引き込む:

- コネクタ・MCP・フック・アプリのインベントリには `workspace-surface-audit`
- 監査が生のリポジトリ真実と耐久性のあるコンテキストを調整する必要があるときには `knowledge-ops`
- 答えが CI・スケジュールワークフロー・issue・PR 自動化に依存する場合は `github-ops`
- 真の問題が兄弟アプリリポジトリでの webhook ファンアウト・キュージョブ・課金燃焼の場合は `ecc-tools-cost-audit`
- ローカルインベントリを現行のプラットフォームサポートやパブリックドキュメントと比較する必要があるときは `research-ops`
- 想定された復旧に頼るのではなく修正後の状態を証明するには `verification-loop`

## 利用するタイミング

- ユーザーが「どんな自動化がある」「何が稼働中」「何が壊れている」「何が重複している」と尋ねる
- タスクが cron ジョブ・GitHub Actions・ローカルフック・MCP サーバ・コネクタ・ラッパー・アプリ統合にまたがる
- ユーザーが別のエージェントシステムから何が移植されたか、ECC 内でまだ何を再構築する必要があるかを知りたい
- ワークスペースに同じことを行う複数の方法が蓄積されており、ユーザーが 1 つの正規レーンを望む

## ガードレール

- ユーザーが明示的に修正を求めていない限り、読み取り専用で始める
- 以下を分離する:
  - 設定済み
  - 認証済み
  - 最近検証済み
  - 古いまたは壊れている
  - 完全に欠落
- スキルや設定がツールを参照しているだけでそれが稼働中だと主張しない
- 証拠テーブルが存在するまで、重複するサーフェスをマージしたり削除したりしない

## ワークフロー

### 1. 実際のサーフェスをインベントリする

理論化する前に現在の稼働中サーフェスを読む:

- repo フックとローカルフックスクリプト
- GitHub Actions とスケジュールワークフロー
- MCP 設定と有効化されたサーバ
- コネクタまたはアプリ裏付けの統合
- ラッパースクリプトとリポジトリ固有の自動化エントリポイント

サーフェスごとにグループ化する:

- ローカルランタイム
- リポジトリ CI / 自動化
- 接続された外部システム
- メッセージング / 通知
- 課金 / 顧客運用
- リサーチ / モニタリング

### 2. 各項目を稼働状態で分類する

サーフェス化された各自動化に以下のマークを付ける:

- 設定済み
- 認証済み
- 最近検証済み
- 古いまたは壊れている
- 欠落

次に問題タイプを分類する:

- 能動的な破損
- 認証停止
- 古いステータス
- 重複または冗長
- 欠落能力

### 3. 証明パスをトレースする

すべての重要な主張を具体的なソースで裏付ける:

- ファイルパス
- ワークフローラン
- フックログ
- 設定エントリ
- 最近のコマンド出力
- 正確な失敗シグネチャ

現在の状態が曖昧なら、監査が完了したフリをするのではなく、そう直接言う。

### 4. keep / merge / cut / fix-next で締める

重複または疑わしい各サーフェスについて、1 つのコールを返す:

- keep
- merge
- cut
- fix next

価値はノイジーな自動化を 1 つの正規 ECC レーンに崩すことにあり、すべての歴史的パスを保存することにはない。

## 出力フォーマット

```text
CURRENT SURFACE
- automation
- source
- live state
- proof

FINDINGS
- active breakage
- overlap
- stale status
- missing capability

RECOMMENDATION
- keep
- merge
- cut
- fix next

NEXT ECC MOVE
- exact skill / hook / workflow / app lane to strengthen
```

## 落とし穴

- 稼働中インベントリを読めるときに記憶から答えない
- 「設定にある」を「動作している」と扱わない
- 壊れた高シグナルパスを命名する前に低価値の冗長性を修正しない
- ユーザーが先にインベントリを求めたなら、タスクをリポジトリ書き直しに広げない

## 検証

- 重要な主張が稼働中の証明パスを引用している
- サーフェス化された各自動化に明確な稼働状態カテゴリのラベルが付いている
- 最終推奨が keep / merge / cut / fix-next を区別している
