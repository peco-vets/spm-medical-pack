# エージェント内省デバッグ

エージェントの実行が繰り返し失敗している、進捗のないままトークンを消費している、同じツールでループしている、または意図したタスクから逸脱している場合にこのスキルを使う。

これはワークフロースキルであり、隠れたランタイムではない。人間にエスカレーションする前にエージェントが体系的に自己デバッグする方法を教える。

## 起動するタイミング

- 最大ツール呼び出し / ループ上限到達による失敗
- 前進のないまま繰り返されるリトライ
- 出力品質を低下させ始めるコンテキストの増大やプロンプトのドリフト
- 期待と現実の間のファイルシステムや環境状態の不一致
- 診断と小さな修正アクションで回復可能と思われるツール障害

## スコープ境界

以下の場合にこのスキルを起動する:
- やみくもにリトライする前に失敗状態をキャプチャする
- 一般的なエージェント固有の失敗パターンを診断する
- コンテインドなリカバリアクションを適用する
- 構造化された人間可読のデバッグレポートを生成する

以下の主要ソースとしては使用しない:
- コード変更後の機能検証 — `verification-loop` を使う
- より狭い ECC スキルが既に存在するフレームワーク固有のデバッグ
- 現在のハーネスが自動的に強制できないランタイムの約束

## 4 フェーズループ

### Phase 1: 失敗のキャプチャ

回復を試みる前に、失敗を正確に記録する。

キャプチャする内容:
- エラータイプ・メッセージ・利用可能ならスタックトレース
- 最後の意味のあるツール呼び出しシーケンス
- エージェントが何をしようとしていたか
- 現在のコンテキスト圧: 繰り返されるプロンプト、巨大な貼り付けログ、重複したプラン、暴走したメモ
- 現在の環境前提: cwd・ブランチ・関連サービス状態・期待されるファイル

最低限のキャプチャテンプレート:

```markdown
## Failure Capture
- Session / task:
- Goal in progress:
- Error:
- Last successful step:
- Last failed tool / command:
- Repeated pattern seen:
- Environment assumptions to verify:
```

### Phase 2: 根本原因診断

何かを変更する前に、失敗を既知のパターンに照合する。

| パターン | 推定される原因 | チェック |
| --- | --- | --- |
| 最大ツール呼び出し / 同じコマンドの繰り返し | ループまたは出口のない観察者パス | 直近 N 回のツール呼び出しの繰り返しを検査する |
| コンテキストオーバーフロー / 推論劣化 | 無制限なメモ・重複プラン・巨大ログ | 重複と低シグナル量について最近のコンテキストを検査する |
| `ECONNREFUSED` / タイムアウト | サービス利用不可または誤ったポート | サービスの健全性・URL・ポートの前提を検証する |
| `429` / クォータ枯渇 | リトライストームまたは欠落したバックオフ | 繰り返し呼び出しを数えリトライ間隔を検査する |
| 書き込み後にファイル欠落 / 古い diff | レース・誤った cwd・ブランチドリフト | パス・cwd・git status・実際のファイル存在を再確認する |
| 「修正」後もテストが失敗する | 仮説が誤っている | 正確な失敗テストを切り分けバグを再導出する |

診断質問:
- これはロジック障害・状態障害・環境障害・ポリシー障害のどれか?
- エージェントは本来の目標を見失い誤ったサブタスクを最適化し始めたか?
- 障害は決定論的か一過性か?
- 診断を検証する最小の可逆アクションは何か?

### Phase 3: コンテインドリカバリ

診断面を変える最小のアクションで回復する。

安全なリカバリアクション:
- 繰り返しリトライを止めて仮説を再陳述する
- 低シグナルコンテキストを削減し、能動的目標・ブロッカー・証拠のみを残す
- 実際のファイルシステム / ブランチ / プロセス状態を再確認する
- タスクを 1 つの失敗コマンド・1 ファイル・1 テストに絞る
- 推測的推論から直接観察に切り替える
- 失敗が高リスクまたは外部ブロックされている場合は人間にエスカレーションする

実際に現在の環境の実ツールを介して行っているのでない限り、「エージェント状態のリセット」や「ハーネス設定の更新」のような根拠のない自動修復アクションを主張しないこと。

コンテインドリカバリのチェックリスト:

```markdown
## Recovery Action
- Diagnosis chosen:
- Smallest action taken:
- Why this is safe:
- What evidence would prove the fix worked:
```

### Phase 4: 内省レポート

次のエージェントまたは人間にとってリカバリが理解可能になるようなレポートで締める。

```markdown
## Agent Self-Debug Report
- Session / task:
- Failure:
- Root cause:
- Recovery action:
- Result: success | partial | blocked
- Token / time burn risk:
- Follow-up needed:
- Preventive change to encode later:
```

## リカバリのヒューリスティクス

以下の介入を順に優先する:

1. 真の目標を 1 文で再陳述する。
2. 記憶を信頼するのではなく世界の状態を検証する。
3. 失敗のスコープを縮小する。
4. 1 つの判別チェックを実行する。
5. その後でのみリトライする。

悪いパターン:
- 同じアクションを微妙に異なる言い回しで 3 回リトライする

良いパターン:
- 失敗をキャプチャする
- パターンを分類する
- 1 つの直接チェックを実行する
- チェックが支持する場合のみプランを変更する

## ECC との統合

- コード変更後はリカバリ後に `verification-loop` を使う。
- 失敗パターンを直感や後のスキルへと昇格させる価値があるときは `continuous-learning-v2` を使う。
- 問題が技術的失敗ではなく意思決定の曖昧さである場合は `council` を使う。
- 衝突するローカル状態やリポドリフトから失敗が来ている場合は `workspace-surface-audit` を使う。

## 出力標準

このスキルが有効な場合、「直しました」だけで終わらないこと。

常に以下を提供する:
- 失敗パターン
- 根本原因の仮説
- リカバリアクション
- 状況が改善されたまたは依然としてブロックされている証拠
