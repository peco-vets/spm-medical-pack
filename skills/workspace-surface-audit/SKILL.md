# Workspace Surface Audit

「このワークスペースとマシンは今実際に何ができるか、次に何を追加または有効化すべきか？」という質問に答えるための読み取り専用監査スキル。

これはセットアップ監査プラグインに対する ECC ネイティブの答え。ユーザーがフォローアップ実装を明示的に依頼しない限り、ファイルを変更しない。

## 使用するタイミング

- ユーザーが「Claude Code をセットアップ」「自動化を推奨」「どのプラグインや MCP を使うべき？」「何が欠けている？」と言う
- スキル、フック、コネクタをさらにインストールする前のマシンまたはリポの監査
- 公式マーケットプレイスプラグインを ECC ネイティブカバレッジと比較
- `.env`、`.mcp.json`、プラグイン設定、接続アプリサーフェスをレビューして欠落ワークフロー層を見つける
- 機能がスキル、フック、エージェント、MCP、または外部コネクタであるべきかを決定

## 譲れないルール

- シークレット値を決して印刷しない。プロバイダ名、機能名、ファイルパス、キーや設定が存在するかどうかのみを表面化する。
- ECC が合理的にサーフェスを所有できるとき、一般的な「別のプラグインをインストール」アドバイスより ECC ネイティブワークフローを推奨。
- 外部プラグインを権威ある製品境界ではなく、ベンチマークとインスピレーションとして扱う。
- 3 つを明確に分離：
  - 現在すでに利用可能
  - 利用可能だが ECC でうまくラップされていない
  - 利用不可で新規統合が必要

## 監査入力

質問にうまく答えるために必要なファイルと設定のみを検査：

1. リポサーフェス
   - `package.json`、ロックファイル、言語マーカー、フレームワーク設定、`README.md`
   - `.mcp.json`、`.lsp.json`、`.claude/settings*.json`、`.codex/*`
   - `AGENTS.md`、`CLAUDE.md`、インストールマニフェスト、フック設定
2. 環境サーフェス
   - アクティブリポと明らかな隣接 ECC ワークスペースの `.env*` ファイル
   - `STRIPE_API_KEY`、`TWILIO_AUTH_TOKEN`、`FAL_KEY` などのキー名のみを表面化
3. 接続ツールサーフェス
   - インストールされたプラグイン、有効なコネクタ、MCP サーバ、LSP、アプリ統合
4. ECC サーフェス
   - すでにニーズをカバーする既存スキル、コマンド、フック、エージェント、インストールモジュール

## 監査プロセス

### フェーズ 1：存在するものをインベントリ

コンパクトなインベントリを生成：

- アクティブハーネスターゲット
- インストールされたプラグインと接続アプリ
- 設定された MCP サーバ
- 設定された LSP サーバ
- キー名で示唆される env バックサービス
- ワークスペースにすでに関連する既存 ECC スキル

サーフェスがプリミティブとしてのみ存在する場合、それを指摘。例：

- 「Stripe は接続アプリ経由で利用可能だが、ECC は請求オペレータスキルがない」
- 「Google Drive は接続されているが、ECC ネイティブな Google Workspace オペレータワークフローがない」

### フェーズ 2：公式およびインストールサーフェスに対してベンチマーク

ワークスペースを以下と比較：

- セットアップ、レビュー、ドキュメント、設計、またはワークフロー品質と重複する公式 Claude プラグイン
- Claude または Codex にローカルインストールされたプラグイン
- ユーザーの現在接続されているアプリサーフェス

名前を単にリストしない。各比較について、答える：

1. それが実際に何をするか
2. ECC にすでにパリティがあるか
3. ECC にプリミティブのみがあるか
4. ECC がワークフローを完全に欠いているか

### フェーズ 3：ギャップを ECC 決定に変える

すべての実ギャップに対して、正しい ECC ネイティブ形状を推奨：

| ギャップタイプ | 推奨 ECC 形状 |
|----------|---------------------|
| 繰り返しオペレータワークフロー | スキル |
| 自動強制または副作用 | フック |
| 専門化された委任ロール | エージェント |
| 外部ツールブリッジ | MCP サーバまたはコネクタ |
| インストール／ブートストラップガイダンス | セットアップまたは監査スキル |

ニーズがインフラ的ではなく運用的なときは、既存ツールをオーケストレートするユーザー向けスキルをデフォルトに。

## 出力フォーマット

この順序で 5 セクションを返す：

1. **現在のサーフェス**
   - 今すぐ使用可能なもの
2. **パリティ**
   - ECC がすでにベンチマークにマッチまたは超えるところ
3. **プリミティブのみのギャップ**
   - ツールは存在するが、ECC にクリーンなオペレータスキルがない
4. **欠落統合**
   - まだ利用不可な機能
5. **次の上位 3-5 の動き**
   - インパクトで順序付けられた具体的な ECC ネイティブ追加

## 推奨ルール

- カテゴリあたり最大 1-2 の最高価値アイデアを推奨。
- 明らかなユーザー意図とビジネス価値を持つスキルを優先：
  - セットアップ監査
  - 請求／顧客 ops
  - issue/プログラム ops
  - Google Workspace ops
  - デプロイ／ops コントロール
- コネクタが企業固有の場合、ユーザーのワークフローに genuinely 利用可能または明らかに有用なときのみ推奨。
- ECC にすでに強いプリミティブがあれば、まったく新しいサブシステムを発明する代わりにラッパースキルを提案。

## 良い結果

- ユーザーが何が接続され、何が欠けており、ECC が次に何を所有すべきかを即座に見える。
- 推奨が別の発見パスなしでリポに実装できるほど具体的。
- 最終的な答えが API ブランドではなくワークフローを中心に整理されている。
