# Terminal Ops

ユーザーが実際のリポ実行を望むときに使う：コマンドの実行、git 状態の検査、CI またはビルドのデバッグ、ナローなフィックスの作成、何が変更され検証されたかを正確に報告。

このスキルは意図的に一般的なコーディングガイダンスより狭い。エビデンスファーストのターミナル実行のためのオペレータワークフロー。

## スキルスタック

関連する場合、これらの ECC ネイティブスキルをワークフローに引き入れる：

- 変更後の正確な証明ステップには `verification-loop`
- 適切な修正がリグレッションカバレッジを必要とするときには `tdd-workflow`
- シークレット、認証、または外部入力が関わるときには `security-review`
- タスクが CI 実行、PR 状態、またはリリース状態に依存するときには `github-ops`
- 検証された結果を永続的なプロジェクトコンテキストにキャプチャする必要があるときには `knowledge-ops`

## 使用するタイミング

- ユーザーが「fix」「debug」「run this」「check the repo」「push it」と言う
- タスクがコマンド出力、git 状態、テスト結果、または検証されたローカル修正に依存する
- 答えがローカル変更、ローカル検証、コミット、プッシュを区別する必要がある

## ガードレール

- 編集前に検査する
- ユーザーが監査／レビューのみを依頼したら読み取り専用のままにする
- 即興のアドホックラッパーよりリポローカルスクリプトとヘルパーを推奨
- 証明コマンドが再実行されるまで修正済みと主張しない
- ブランチが実際にアップストリームに移動するまでプッシュ済みと主張しない

## ワークフロー

### 1. 作業面を解決する

確定する：

- 正確なリポパス
- ブランチ
- ローカル diff 状態
- 要求されたモード：
  - inspect
  - fix
  - verify
  - push

### 2. 失敗面を最初に読む

何も変更する前に：

- エラーを検査
- ファイルまたはテストを検査
- git 状態を検査
- 盲目的に再読み込みする前に、すでに提供されたログまたはコンテキストを使う

### 3. 修正を狭く保つ

一度に 1 つの支配的な失敗を解決する：

- 最小の有用な証明コマンドを先に使う
- ローカル失敗が対処された後にのみ、より大きなビルド／テストパスにエスカレートする
- コマンドが同じ署名で失敗し続けるなら、広範な再試行を停止しスコープを狭める

### 4. 正確な実行状態を報告する

正確なステータス語を使う：

- inspected
- changed locally
- verified locally
- committed
- pushed
- blocked

## 出力フォーマット

```text
SURFACE
- repo
- branch
- requested mode

EVIDENCE
- failing command / diff / test

ACTION
- what changed

STATUS
- inspected / changed locally / verified locally / committed / pushed / blocked
```

## 落とし穴

- ライブリポ状態が読めるときに古いメモリから作業しない
- ナローなフィックスをリポワイドのチャーンに広げない
- 破壊的 git コマンドを使わない
- 無関係なローカル作業を無視しない

## 検証

- レスポンスが証明コマンドまたはテストを名指す
- git 関連作業はリポパスとブランチを名指す
- 任意のプッシュ主張はターゲットブランチと正確な結果を含む
