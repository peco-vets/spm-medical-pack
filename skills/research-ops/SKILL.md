# Research Ops

ユーザーが最新のものを調査したい、選択肢を比較したい、人物や企業をエンリッチしたい、繰り返しのルックアップをモニタリング対象のワークフローに変えたい場合に使う。

これはリポジトリのリサーチスタック周りのオペレータラッパーである。`deep-research`、`exa-search`、`market-research` の代替ではなく、それらをいつどう組み合わせて使うかを教えるものである。

## スキルスタック

関連する場合、これらの ECC ネイティブスキルをワークフローに引き入れる：

- 高速な現在の Web 発見には `exa-search`
- 引用付きマルチソース合成には `deep-research`
- 最終結果が推奨またはランク付き意思決定であるべきときには `market-research`
- 一般的な調査ではなく人物／企業のターゲティングが目的のときには `lead-intelligence`
- 結果を後で耐久的なコンテキストに保存すべきときには `knowledge-ops`

## 使用するタイミング

- ユーザーが「research」「look up」「compare」「who should I talk to」「what's the latest」と言う
- 答えが現在の公開情報に依存する
- ユーザーがすでにエビデンスを提供し、新鮮な推奨に取り込みたい
- タスクが十分に繰り返されるので、単発のルックアップではなくモニタにすべき可能性がある

## ガードレール

- 新鮮な検索が安価なときに、古い記憶から現在の質問に答えない
- 以下を分離する：
  - ソース付き事実
  - ユーザー提供エビデンス
  - 推論
  - 推奨
- 答えがすでにローカルコード／ドキュメントにあるなら、重量級リサーチパスを立ち上げない

## ワークフロー

### 1. ユーザーがすでに与えたものから始める

提供された資料を以下に正規化する：

- すでにエビデンス付きの事実
- 検証が必要
- オープンな質問

ユーザーがすでにモデルの一部を構築している場合、解析をゼロから再開しない。

### 2. 依頼を分類する

検索する前に正しいレーンを選ぶ：

- 素早い事実回答
- 比較または意思決定メモ
- リード／エンリッチメントパス
- 繰り返しモニタリング候補

### 3. 最も軽い有用なエビデンスパスを先に取る

- 高速発見には `exa-search` を使う
- 合成または複数ソースが重要なときには `deep-research` にエスカレートする
- 結果が推奨で終わるべきときには `market-research` を使う
- 真の依頼がターゲットランキングまたは温かいパスの発見である場合、`lead-intelligence` に引き継ぐ

### 4. 明示的なエビデンス境界で報告する

重要なクレームについては、以下のいずれかを述べる：

- ソース付き事実
- ユーザー提供コンテキスト
- 推論
- 推奨

鮮度に敏感な回答には具体的な日付を含める。

### 5. タスクを手動のままにすべきかを判断する

ユーザーが同じリサーチ質問を繰り返し聞く可能性が高い場合、それを明示的に述べ、同じ手動検索を永遠に繰り返すのではなく、モニタリングまたはワークフロー層を推奨する。

## 出力フォーマット

```text
QUESTION TYPE
- factual / comparison / enrichment / monitoring

EVIDENCE
- sourced facts
- user-provided context

INFERENCE
- what follows from the evidence

RECOMMENDATION
- answer or next move
- whether this should become a monitor
```

## 落とし穴

- ラベル付けせずにソース付き事実に推論を混ぜない
- ユーザー提供エビデンスを無視しない
- ローカルリポコンテキストで答えられる質問に重量級リサーチレーンを使わない
- 日付なしで鮮度に敏感な回答を出さない

## 検証

- 重要なクレームはエビデンスタイプでラベル付けされている
- 鮮度に敏感な出力には日付が含まれる
- 最終推奨は実際に使われたリサーチモードと一致する
