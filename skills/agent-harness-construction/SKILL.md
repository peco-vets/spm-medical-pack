# エージェントハーネス構築

エージェントが計画を立て、ツールを呼び出し、エラーから回復し、完了に収束する仕組みを改善する際にこのスキルを使用する。

## コアモデル

エージェントの出力品質は以下によって制約される:
1. アクション空間の品質
2. 観測の品質
3. リカバリの品質
4. コンテキスト予算の品質

## アクション空間の設計

1. 安定で明示的なツール名を使う。
2. 入力はスキーマファーストで狭く保つ。
3. 決定論的な出力形状を返す。
4. 分離不可能な場合を除き、なんでもツールは避ける。

## 粒度ルール

- ハイリスクな操作 (デプロイ・マイグレーション・権限) にはマイクロツールを使う。
- 一般的な編集/読み込み/検索ループにはミディアムツールを使う。
- ラウンドトリップのオーバーヘッドが支配的コストである場合のみマクロツールを使う。

## 観測の設計

すべてのツール応答に以下を含める:
- `status`: success|warning|error
- `summary`: 1 行の結果
- `next_actions`: 実行可能なフォローアップ
- `artifacts`: ファイルパス / ID

## エラーリカバリ契約

すべてのエラーパスに以下を含める:
- 根本原因のヒント
- 安全なリトライ指示
- 明示的な停止条件

## コンテキスト予算

1. システムプロンプトは最小限かつ不変に保つ。
2. 大規模なガイダンスはオンデマンドでロードされるスキルへ移す。
3. 長文書をインライン化するよりファイルへの参照を優先する。
4. 任意のトークン閾値ではなく、フェーズ境界で圧縮する。

## アーキテクチャパターンガイダンス

- ReAct: 経路が不確実な探索的タスクに最適。
- Function-calling: 構造化された決定論的フローに最適。
- ハイブリッド (推奨): ReAct プランニング + 型付きツール実行。

## ベンチマーキング

以下を追跡する:
- 完了率
- タスク当たりのリトライ数
- pass@1 と pass@3
- 成功タスク当たりのコスト

## アンチパターン

- 重複するセマンティクスを持つツールが多すぎる。
- リカバリヒントのない不透明なツール出力。
- 次のステップを示さないエラーのみの出力。
- 無関係な参照によるコンテキストの過負荷。
