# Rules

## 必ず守ること (Must Always)
- ドメインタスクは専門エージェントに委任する。
- 実装前にテストを書き、クリティカルパスを検証する。
- 入力を検証し、セキュリティチェックを損なわない。
- 共有状態の変更よりイミュータブルな更新を選ぶ。
- 新規パターンを生み出す前に、確立されたリポジトリパターンに従う。
- コントリビューションは焦点を絞り、レビュー可能で、十分に説明されたものに保つ。

## 絶対にしないこと (Must Never)
- API キー、トークン、シークレット、絶対パスやシステムパスなどの機密データを出力に含めない。
- 未テストの変更を提出しない。
- セキュリティチェックや検証フックをバイパスしない。
- 明確な理由無く既存機能を重複させない。
- 該当テストスイートを確認せずにコードをリリースしない。

## Agent 形式
- エージェントは `agents/*.md` に配置する。
- 各ファイルは `name`, `description`, `tools`, `model` を持つ YAML frontmatter を含む。
- ファイル名は小文字とハイフンで構成し、エージェント名と一致させる。
- description はエージェントを呼び出すタイミングを明確に伝えること。

## Skill 形式
- スキルは `skills/<name>/SKILL.md` に配置する。
- 各スキルは `name`, `description`, `origin` を持つ YAML frontmatter を含む。
- ファーストパーティスキルは `origin: ECC`、インポート・コミュニティスキルは `origin: community` とする。
- スキル本文は実用ガイダンス、テスト済みの例、明確な "When to Use" セクションを含めること。

## Hook 形式
- フックは matcher 駆動の JSON 登録と、shell または Node エントリポイントを使う。
- matcher は広範なキャッチオールではなく、具体的にする。
- ブロック挙動が意図的な場合にのみ `exit 1` を返す。それ以外は `exit 0` を返す。
- エラーメッセージと情報メッセージは実行可能なものにする。

## コミットスタイル
- `feat(skills):`, `fix(hooks):`, `docs:` のような Conventional Commits を使う。
- 変更はモジュラーに保ち、PR サマリでユーザー向け影響を説明する。
